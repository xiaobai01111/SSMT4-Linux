# ssmt4-bridge — Windows Middleware for 3DMigoto Injection

A C++ Windows executable that runs inside a Proton container alongside the game. It handles 3DMigoto DLL deployment, INI configuration, mod optimization, game-specific initialization, and DLL injection.

## Architecture

```
┌─────────────────────────────────┐
│  Linux Tauri Launcher (Rust)    │
│  - UI, package downloads        │
│  - Generates bridge-config.json │
│  - Launches bridge in Proton    │
│  - Reads stdout JSON messages   │
└──────────┬──────────────────────┘
           │  Same Proton container
           ▼
┌─────────────────────────────────┐
│  ssmt4-bridge.exe (C++/Win32)   │
│  - Reads bridge-config.json     │
│  - Updates d3dx.ini             │
│  - Optimizes Mods/ShaderFixes   │
│  - Game-specific init (SQLite)  │
│  - Deploys & verifies DLLs      │
│  - Hook or Direct DLL injection │
│  - Reports via stdout JSON      │
└──────────┬──────────────────────┘
           │
           ▼
┌─────────────────────────────────┐
│  Game Process (e.g. WuWa)       │
│  + d3d11.dll (3DMigoto)         │
│  + Mods loaded                  │
└─────────────────────────────────┘
```

## Key Design Principles

1. **Fully data-driven** — All paths, game exe, container info come from `bridge-config.json` generated by the Linux launcher. Zero hardcoded values in C++.
2. **Same container** — Bridge runs in the identical Proton prefix/environment as the game, sharing process namespace for DLL injection.
3. **Per-game isolation** — Each game has its own `importer_folder`, `d3dx.ini`, `Mods/`, `ShaderFixes/`. Modifying one game never affects another.

## Build

### Prerequisites

- **MinGW-w64 cross-compiler**: `x86_64-w64-mingw32-g++`
- **CMake** ≥ 3.16
- **Make** (GNU Make)

#### Install on Arch Linux

```bash
sudo pacman -S mingw-w64-gcc cmake make
```

#### Install on Ubuntu / Debian

```bash
sudo apt install gcc-mingw-w64-x86-64 g++-mingw-w64-x86-64 cmake make
```

#### Install on Fedora

```bash
sudo dnf install mingw64-gcc mingw64-gcc-c++ cmake make
```

### Quick Build

```bash
cd Windows/
./build.sh              # Release build (default)
./build.sh Debug        # Debug build with symbols
```

### Manual Build

```bash
cd Windows/
mkdir -p build && cd build
cmake .. -DCMAKE_TOOLCHAIN_FILE=../toolchain-mingw64.cmake -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
```

### Output

- `build/ssmt4-bridge.exe` — Windows PE executable (~200 KB)
- Statically links cJSON and sqlite3 (no external DLL dependencies except Win32 system libs)

### Deployment

The built `ssmt4-bridge.exe` must be placed at `<SSMT4-Linux>/Windows/ssmt4-bridge.exe`.
The Tauri launcher automatically locates and runs it from there inside the Proton container.

## Tauri Integration

The Rust side (`src-tauri/src/commands/bridge.rs`) provides:

- `build_bridge_config()` — Generates config from game settings
- `write_bridge_config()` — Writes JSON to `Cache/bridge/bridge-config.json`
- `run_bridge()` — Launches bridge.exe in the same Proton container, parses stdout JSON, emits Tauri events

## Communication Protocol

Bridge → Launcher (stdout, one JSON per line):

| Type | Fields | Description |
|------|--------|-------------|
| `status` | `message` | Progress status text |
| `progress` | `stage`, `current`, `total` | Numeric progress |
| `warning` | `message` | Non-fatal warning |
| `error` | `code`, `message` | Fatal error |
| `inject_result` | `method`, `success`, `pid` | Injection outcome |
| `log` | `level`, `message` | Debug/info log |
| `done` | `success` | Final completion signal |

## Adding New Games

1. Create a new initializer class (e.g. `ZZMIInitializer`) extending `GameInitializer`
2. Register it in `GameInitializer::create()` factory
3. Add game-specific settings to `bridge-config.json` under the importer key
4. The frontend populates game-specific config — no C++ changes needed for simple config-driven games
