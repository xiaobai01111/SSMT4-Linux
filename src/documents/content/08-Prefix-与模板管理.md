# Prefix 与模板管理

最后更新：2026-03-08

这篇文档解释 SSMT4-Linux 里 Prefix 的真实结构、路径选择、迁移逻辑和模板体系。

如果你只把 Prefix 理解成“一个 Wine 文件夹”，后面很多现象都会看错：

- 为什么游戏设置里已经显示“已创建”，但运行时仍然可能缺组件；
- 为什么有些 Prefix 在游戏目录里，有些旧 Prefix 在 `~/.config/ssmt4/prefixes`；
- 为什么“模板”有两套，一套是 Prefix 模板，另一套是游戏配置模板；
- 为什么删除、迁移、重命名 Prefix 时项目会格外保守。

## 1. 先分清 Prefix 目录、`pfx` 和 `prefix.json`

在当前项目里，一个完整的 Prefix 相关目录通常至少包含两部分：

- `prefix.json`
- `pfx/`

它们不是一回事。

### 1.1 `prefix.json`

这是 SSMT4-Linux 自己维护的 Prefix 配置文件。

它主要记录：

- 当前游戏绑定的 Wine/Proton 版本 ID
- Prefix 架构
- DXVK / VKD3D 开关与版本提示
- 已安装的运行时组件列表
- 每个 Prefix 的环境变量覆盖
- 来源模板 ID
- Proton 运行参数

它是“项目配置层”，不是 Wine 自己生成的注册表或系统文件。

### 1.2 `pfx/`

这是 Wine/Proton 实际使用的 Prefix 根目录。

里面才会有这些内容：

- `drive_c`
- `system.reg`
- `user.reg`
- `dosdevices`
- Windows 程序运行过程中需要的真实目录结构

可以把它理解成“真正的 Windows 容器内容”。

### 1.3 Prefix 目录本身

当前项目把 Prefix 顶层目录当成“容器包裹层”。

也就是说：

- 顶层目录负责同时容纳 `prefix.json`、`pfx/`、以及某些附加目录；
- `pfx/` 才是 Wine/Proton 真正消费的 Prefix；
- 某些运行期辅助目录也会挂在顶层，比如隔离 home 时的 `sandbox-home/`。

## 2. Prefix 路径是怎样决定的

Prefix 路径不是固定写死在一个地方，而是分优先级解析。

### 2.1 优先使用游戏目录下的 `prefix/`

如果项目能从当前游戏配置推导出游戏根目录，那么首选路径总是：

- `<游戏根目录>/prefix`

项目推导游戏根目录时，优先看：

- `gameFolder`
- 回退到 `gamePath`
- 必要时结合游戏预设里的 `defaultFolder` 再向上推断

这意味着当前实现倾向于把 Prefix 和游戏安装目录放在一起，而不是一律塞进全局配置目录。

### 2.2 无法推导游戏根目录时，回退到旧式全局 Prefix 目录

如果项目没法从配置里可靠推导出游戏根目录，就会回退到旧位置：

- `~/.config/ssmt4/prefixes/<游戏ID>`

这个目录仍然有兼容价值，因为：

- 老版本留下的 Prefix 可能还在这里；
- 某些配置不完整时，项目只能先回退到这里。

### 2.3 旧 Prefix 会尝试自动迁移

如果项目已经能推导出新的首选路径，但检测到旧 Prefix 还在全局目录里，就会尝试迁移。

当前实现的迁移逻辑是：

1. 先尝试 `rename`
2. 如果跨磁盘或挂载点导致 `rename` 失败，再回退到“复制 + 删除旧目录”
3. 如果迁移失败，本轮进程内会记住失败结果，避免反复重试
4. 迁移失败后，项目会在新位置继续创建新的 Prefix

所以你看到“旧 Prefix 没了，游戏目录下出现了新的 prefix/”，很多时候不是丢失，而是迁移策略在工作。

### 2.4 旧别名 Prefix 也会参与选择

项目在寻找旧 Prefix 时，不只看当前标准游戏 ID，还会看历史别名。

并且会按目录内容打分，例如：

- 有没有 `prefix.json`
- 有没有 `pfx/`
- `pfx/` 里有没有 `.dxvk-version`

分数更高的目录更可能被当作“最佳旧 Prefix”。

## 3. 创建 Prefix 不等于已经初始化完成

这是最容易误解的一点。

### 3.1 `create_prefix` 只做基础目录与配置落盘

当前后端的 `create_prefix` 本质上只做这些事情：

- 创建 Prefix 顶层目录
- 创建 `pfx/` 目录
- 写入 `prefix.json`

这一步并不保证：

- `drive_c/windows` 已经完整生成
- 注册表已经由 Wine/Proton 初始化完成
- DXVK / VKD3D 已经安装
- 运行时组件已经装好

### 3.2 真正的 Prefix 初始化通常发生在后续运行期

实际的 Wine Prefix 内容，往往是在后续启动或安装组件时才逐步补齐。

所以你在游戏设置里看到：

- `✓ 已创建`

它表达的是“SSMT4-Linux 这一层的 Prefix 目录已存在”，不等于“这个 Prefix 已经是一个完全可运行的 Windows 环境”。

### 3.3 首次启动时还可能发生自动补全

在启动链里，如果 `prefix.json` 加载失败或不存在，项目会自动创建默认 Prefix 配置。

随后还会做这些事情：

- 检查 DXVK / VKD3D 是否已安装
- 尝试给 Prefix 补 CJK 字体
- 记录运行时状态到游戏日志窗口

所以“首次启动之后 Prefix 看起来突然更完整了”是正常现象。

## 4. 当前 Prefix 里保存了哪些重要信息

当前 `PrefixConfig` 里最值得理解的字段有这些。

### 4.1 `wine_version_id`

这是当前游戏绑定的 Wine/Proton 版本 ID。

它不是路径，而是一个逻辑 ID。真正启动时，项目会重新扫描本地 Wine/Proton 列表，再按这个 ID 找到对应版本。

所以：

- 改了本机版本目录结构后，这个 ID 可能失效；
- 失效时启动器会在运行环境检查时报错。

### 4.2 `dxvk` 与 `vkd3d`

这两个字段表达的是：

- 是否启用
- 期望版本提示

它们不等于“实际已安装”。真正是否安装成功，运行前还会再次检测 `pfx/` 里的 DLL 和版本标记。

### 4.3 `installed_runtimes`

这是项目记录的“已安装运行时组件列表”。

它反映的是 SSMT4-Linux 自己执行安装操作后的记录，不是自动从整个 Prefix 环境做深度逆向扫描得出的完整真相。

所以它更像“项目的安装历史状态”，不是万能法官。

### 4.4 `env_overrides`

这是 Prefix 级别的环境变量覆盖。

它和全局环境变量、自定义启动环境不同，优先级更贴近当前游戏对应 Prefix 的运行需求。

这类字段适合放：

- `WINEDLLOVERRIDES`
- 某些按 Prefix 固定的兼容性变量

但它也会增加排障复杂度，所以不要在没有日志基线的情况下乱堆。

### 4.5 `proton_settings`

这里保存的是单游戏运行相关的 Proton 设置，比如：

- `use_umu_run`
- `use_pressure_vessel`
- `sandbox_enabled`
- `sandbox_isolate_home`
- `proton_media_use_gst`
- `proton_enable_wayland`
- `proton_no_d3d12`
- `dxvk_hud`
- `dxvk_async`
- `dxvk_frame_rate`
- `custom_env`

所以当前项目里很多“运行环境”开关，其实最后都会落回 Prefix 配置层。

### 4.6 `template_id`

如果这个 Prefix 是从模板创建的，这里会记录模板 ID。

它的意义主要是“来源追踪”，不是“后续自动双向同步”。

也就是说：

- 你从模板创建了 Prefix；
- 之后手动改了 Prefix；
- 模板本身不会自动跟着改。

## 5. Prefix 生命周期里的几个关键动作

### 5.1 自动创建

这些情况下，Prefix 可能被自动创建：

- 你在设置运行环境时保存了游戏 Wine 配置，但当前 Prefix 不存在；
- 启动游戏时发现 `prefix.json` 缺失或加载失败；
- 某些基于模板的创建逻辑被调用。

### 5.2 删除

项目允许删除 Prefix，但删除前做了白名单校验。

它不会随便删任何你传进去的路径，而是要确认：

- 目标路径确实在允许的游戏根目录内，且目录名像 `prefix`
- 或者目标路径在兼容旧结构的 `~/.config/ssmt4/prefixes` 下
- 并且不会误删整个 `prefixes` 根目录

这是一层故意保守的安全边界。

### 5.3 迁移与重命名

除了前面说的“旧 Prefix 自动迁移到游戏目录”，当前项目还有一类迁移：

- 游戏标识规范化迁移

当旧的游戏别名需要统一成新的 canonical ID 时，项目也可能：

- 重命名旧的全局 Prefix 目录
- 同步重写部分配置里的旧路径引用

这主要服务于历史兼容，不是日常普通用户需要频繁手动触发的流程。

## 6. Prefix 和运行时组件之间的真实关系

### 6.1 DXVK / VKD3D 安装位置在 `pfx/`

无论 UI 上怎么叫“安装到当前游戏”，底层真正装入的是：

- 当前游戏 Prefix 的 `pfx/`

所以图形层组件的安装对象从来不是“某个全局游戏配置项”，而是“这个 Prefix 对应的 Windows 容器”。

### 6.2 运行时组件安装依赖 Prefix 当前绑定的 Wine 版本

安装运行时组件时，项目会：

1. 读取 Prefix 配置
2. 找到当前绑定的 Wine/Proton 版本
3. 用它来执行运行时安装
4. 成功后把组件名记入 `installed_runtimes`

如果这一步找不到对应 Wine 版本，就会直接报错。

所以“先选运行器，再装运行时”在这里不是建议，而是实现前提。

### 6.3 CJK 字体是按 Prefix 补的

当前项目在启动时会尝试把系统里的 CJK 字体链接到 Prefix 的字体目录，并在 `system.reg` 中补写字体替换项。

这里要注意两个边界：

- 如果 `drive_c/windows/Fonts` 还不存在，说明 Prefix 还没被真正初始化，这一步会直接跳过；
- 这一步是按 Prefix 做的，不是全局系统级魔法。

所以中文乱码问题经常会出现一种现象：

- 第一次启动前还没完全准备好；
- 启动过一次后，字体补全才开始生效。

### 6.4 `sandbox-home/` 不属于 Wine Prefix 正文

如果启用了运行时沙盒并隔离 `HOME`，项目会在 Prefix 顶层目录下创建：

- `sandbox-home/`

然后把运行进程的 `HOME` 指向这里。

它的作用是运行时隔离，不是 Wine Prefix 的核心系统结构。

所以你看到 Prefix 顶层多了这个目录，不要误以为 Prefix 被损坏了。

## 7. 当前项目里其实有两套“模板”

这点必须分清。

## 7.1 Prefix 模板

Prefix 模板描述的是“要创建一个什么样的 Prefix 配置”。

它保存的位置是：

- `~/.config/ssmt4/prefixes/_templates/*.json`

也就是配置目录下的模板 JSON 文件，而不是数据目录。

它包含的核心内容包括：

- 架构
- DXVK / VKD3D 配置
- 推荐的 Proton 变体
- 所需运行时列表
- 环境变量覆盖
- Proton 运行参数

### 7.1.1 Prefix 模板创建时会带哪些字段

当项目用模板创建 Prefix 时，会把模板里的这些内容写进新 Prefix：

- `arch`
- `dxvk`
- `vkd3d`
- `env_overrides`
- `proton_settings`
- `template_id`

但它不会直接把一切都无脑复制过去。

例如当前实现里：

- `wine_version_id` 会留空
- `installed_runtimes` 会从空列表开始

这说明 Prefix 模板更偏向“配置蓝图”，而不是完整的环境镜像。

### 7.1.2 当前前端边界

后端和 API 已经有这些能力：

- 列出 Prefix 模板
- 保存 Prefix 模板
- 按模板创建 Prefix

另外，后端内部还存在“从现有 Prefix 导出模板”的辅助函数。

但要注意当前边界：

- 主界面还没有一个成熟的 Prefix 模板管理页；
- 游戏设置里主要展示 Prefix 状态和运行环境，而不是完整的模板工作流；
- 导出 Prefix 为模板的流程目前也没有公开成稳定的前端入口。

所以当前 Prefix 模板更像“已经有底层能力、但前端仍在完善中的高级能力”。

## 7.2 游戏配置模板

游戏配置模板是另一套东西，它描述的是：

- 某个游戏在 `Games` 目录下的一套配置骨架

它保存的位置是：

- `<dataDir>/GameTemplates/<模板目录>/`

这和 Prefix 模板完全不是同一个目录体系。

### 7.2.1 一个游戏配置模板通常包含什么

当前实现会把它当成一个目录来处理，至少会看：

- `Config.json`
- 可选的 `Icon.png`

其中 `Config.json` 里的这些字段尤其重要：

- `LogicName`
- `GamePreset`
- `DisplayName`

项目会用这些字段来判断模板属于哪个游戏、显示什么名字、是否已经存在同一逻辑游戏配置。

### 7.2.2 游戏配置模板从哪里用

这套模板当前有明确的前端入口：

- `游戏库`
- 空白处右键菜单的“导入配置”
- 导入弹窗
- “打开模板文件夹”

所以你在游戏库里导入的是“游戏配置模板”，不是 Prefix 模板。

### 7.2.3 覆盖导入会发生什么

如果同一逻辑游戏已经存在：

- 前端会弹确认；
- 确认覆盖后，旧的游戏目录会先被删除；
- 然后模板目录会被完整复制到 `Games` 目录。

这一步操作的是游戏配置目录，不是 Prefix 目录。

所以：

- 导入一个游戏配置模板，不会自动复制你原来的 Prefix；
- 删除旧游戏配置目录，也不等于删除对应 Prefix。

## 8. 推荐工作流

### 8.1 新装游戏后的稳定流程

1. 先导入或创建游戏配置。
2. 配好主程序路径与运行环境。
3. 启动一次，让 Prefix 进入真实初始化阶段。
4. 再安装 DXVK、VKD3D 和运行时组件。
5. 最后才考虑环境变量、沙盒隔离和实验性功能。

### 8.2 需要复用一套容器配置时

如果你想复用的是：

- DXVK / VKD3D 开关
- 架构
- 环境变量覆盖
- Proton 运行参数

那么应优先考虑 Prefix 模板。

如果你想复用的是：

- 游戏条目
- 预设
- 显示名
- 图标
- Config.json 骨架

那么应优先考虑游戏配置模板。

### 8.3 出现问题时不要立刻删 Prefix

先判断问题属于哪一层：

- 路径或配置骨架问题：先看游戏配置
- 运行时、图形层、字体、组件问题：先看 Prefix
- 实验性功能问题：先回到最小基线

很多问题不需要删 Prefix，只需要：

- 改回正确运行器
- 补装图形层
- 补装运行时
- 暂时关闭实验性能力

## 9. 常见误区

- 误区一：游戏设置里显示 Prefix 已创建，就说明容器完整可用。不是，很多实际内容要等首次运行或组件安装后才会补齐。
- 误区二：导入游戏模板会连 Prefix 一起导入。不是，游戏配置模板和 Prefix 模板是两套系统。
- 误区三：Prefix 模板等于完整克隆现有 Prefix。不是，当前更像“配置蓝图”，不是完整环境快照。
- 误区四：删掉游戏配置就等于删掉 Prefix。不一定，二者目录体系不同。
- 误区五：Prefix 一定只在 `~/.config/ssmt4/prefixes` 里。不是，当前实现优先把 Prefix 放在游戏目录下的 `prefix/`。
