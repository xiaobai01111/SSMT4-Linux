# DXVK / VKD3D 下载、管理与使用

最后更新：2026-03-08

本文解释 SSMT4-Linux 中 DXVK 和 VKD3D 的真实工作方式，而不是只告诉你“去哪里点下载”。

如果你理解这条链路，就会知道：

- 为什么 `设置 -> DXVK 管理` / `设置 -> VKD3D 管理` 和 `游戏设置 -> 运行环境` 是两层不同功能；
- 为什么“已经下载到本地缓存”不等于“当前游戏已经生效”；
- 为什么切换 DXVK 或 VKD3D 有时能解决黑屏、闪退和图形异常，有时却完全无效；
- 为什么某些问题属于 Vulkan 驱动、Prefix 污染、3DMigoto 叠加或游戏自身资源回收，而不是 DXVK/VKD3D 本身。

## 1. 先理解三层职责

当前项目中的图形兼容层管理可以分成三层：

### 1.1 全局缓存层：`设置 -> DXVK 管理` / `设置 -> VKD3D 管理`

这一层负责：

- 获取远程可下载版本；
- 下载压缩包到本地工具缓存；
- 自动解压到本地缓存目录；
- 删除本地缓存中的压缩包和已解压目录。

这一层不直接改动任何游戏 Prefix。

### 1.2 单游戏安装层：`游戏设置 -> 运行环境`

这一层负责：

- 查看当前游戏 Prefix 是否已安装 DXVK；
- 查看当前游戏 Prefix 是否已安装 VKD3D；
- 从本地缓存中选择具体版本；
- 将选中的版本复制到当前游戏 Prefix；
- 从当前游戏 Prefix 卸载 DXVK / VKD3D。

这一层才真正决定“当前游戏到底装了什么”。

### 1.3 启动检测层：`启动游戏`

启动时，项目会再次检测：

- 当前 Prefix 中是否真的有 DXVK DLL；
- 当前 Prefix 中是否真的有 VKD3D DLL；
- Prefix 配置中的提示值和实际检测结果是否一致；
- Vulkan 和 GPU 选择相关环境是否会影响 DXVK / VKD3D。

所以一句话概括：

- `设置页` 管缓存；
- `运行环境页` 管当前游戏 Prefix；
- `启动器` 管最后一次运行前校验和环境变量注入。

## 2. DXVK 和 VKD3D 分别解决什么

这两者不是同一个组件，也不应该混为一谈。

### 2.1 DXVK

DXVK 主要负责：

- Direct3D 9 -> Vulkan
- Direct3D 10 -> Vulkan
- Direct3D 11 -> Vulkan
- DXGI 相关接口替换

项目实际安装的 DXVK DLL 主要是：

- `d3d9.dll`
- `d3d10core.dll`
- `d3d11.dll`
- `dxgi.dll`

### 2.2 VKD3D-Proton

VKD3D-Proton 主要负责：

- Direct3D 12 -> Vulkan

项目实际安装的 VKD3D DLL 主要是：

- `d3d12.dll`
- `d3d12core.dll`
- `dxil.dll`

因此：

- DX11 游戏主要看 DXVK；
- D3D12 游戏主要看 VKD3D-Proton；
- 某些游戏既会受到 DXVK 设置影响，也会受到 VKD3D 和 Proton 开关影响。

## 3. 本地缓存目录放在哪里

项目把图形兼容层缓存放在工具目录下，而不是直接写进任意游戏目录：

- DXVK 缓存目录：`<dataDir>/tools/dxvk/`
- VKD3D 缓存目录：`<dataDir>/tools/vkd3d/`

在这些目录里，项目会同时识别两类内容：

- 压缩包归档
- 已解压目录

因此“本地已缓存”并不一定等于“已经解压”，而“已经解压”也不等于“已经安装到某个 Prefix”。

## 4. DXVK 管理页真正管理的是什么

`设置 -> DXVK 管理` 并不是一个固定下载源页面，它管理的是一套“本地缓存 + 远程来源 + 变体”系统。

### 4.1 DXVK 有变体概念

当前项目的 DXVK 远程来源不是写死在一个函数里，而是来自模块化数据定义。

默认可见的变体至少包括：

- `DXVK`
- `DXVK-GPLAsync`
- `DXVK-Async`
- `DXVK-Sarek`
- `DXVK-Sarek-Async`

这些变体来自 `Data-parameters/dxvk/` 目录中的模块化配置，而不是只有官方 DXVK 一个仓库。

### 4.2 DXVK 远程列表如何获取

DXVK 远程版本获取逻辑会：

- 读取变体定义；
- 按不同 provider 拉取版本；
- 支持 GitHub Releases 和特定 GitLab tree 来源；
- 标记哪些版本已经在本地缓存中存在；
- 某个来源失败时尝试使用上次成功获取的缓存结果。

这意味着：

- DXVK 管理页看到的是“多来源聚合结果”；
- 某个来源临时失败，不一定会让整个页面完全空掉；
- 使用 GitHub token 可以缓解 GitHub API 限流问题。

### 4.3 DXVK 本地缓存如何识别

项目扫描本地 DXVK 缓存时，会按“变体 + 版本”聚合，并且做一个重要处理：

- 同一版本同时存在压缩包和已解压目录时，优先把它视为“已解压版本”；
- 只要目录中存在 `x64` 或 `x32` 结构，就会被视为可直接安装的解压版本。

所以你在设置页看到的“本地已缓存”其实是整理后的结果，不是简单列目录。

## 5. VKD3D 管理页真正管理的是什么

`设置 -> VKD3D 管理` 的结构比 DXVK 更简单，但不是“没有逻辑”。

### 5.1 VKD3D 当前是官方单来源体系

当前 VKD3D 远程版本来自官方 `HansKristian-Work/vkd3d-proton` Releases。

和 DXVK 不同，当前项目没有为 VKD3D 做多变体目录体系，而是：

- 获取官方 release；
- 过滤预发布；
- 只接受符合 `vkd3d-proton-*.tar.zst` 的归档；
- 与本地缓存版本做比对。

### 5.2 VKD3D 本地缓存如何识别

项目扫描 `tools/vkd3d/` 时，会把同一版本的压缩包和解压目录归并成一个版本项。

并且会检查这些结构来判断是否已解压：

- `x64`
- `x86`
- `x32`
- 或更深层级里能找到这些架构目录

这意味着 VKD3D 的本地缓存识别比“只看根目录是否存在 x64”更宽松。

## 6. 下载一个版本时，项目实际做了什么

无论是 DXVK 还是 VKD3D，设置页的下载都不是“浏览器下载一个文件”这么简单。

### 6.1 DXVK 下载流程

大致流程是：

1. 根据所选变体和版本计算归档名称。
2. 根据变体来源生成下载 URL。
3. 下载到 `<dataDir>/tools/dxvk/`。
4. 归档不存在时才真正发起下载。
5. 若对应解压目录不存在，则自动解压。
6. 最后把该版本标记为“本地已缓存”。

### 6.2 VKD3D 下载流程

大致流程是：

1. 计算 `vkd3d-proton-{version}.tar.zst` 归档名。
2. 从官方 Release URL 下载到 `<dataDir>/tools/vkd3d/`。
3. 若未解压，则自动解压。
4. 最后把版本标记为“本地已缓存”。

### 6.3 这一层不会做的事

设置页下载不会：

- 自动改动任意游戏 Prefix；
- 自动覆盖某个游戏当前正在使用的 DLL；
- 自动切换某个游戏的运行环境配置。

所以“下载完成”只意味着缓存准备好了，不意味着游戏已经用上。

## 7. 安装到当前游戏 Prefix 时，项目具体做了什么

当你在 `游戏设置 -> 运行环境` 中点击“应用 / 切换版本”时，项目会把缓存中的 DLL 真正复制到 Prefix。

### 7.1 DXVK 安装流程

DXVK 安装时会：

- 确认缓存归档存在，不存在就补下载；
- 确认解压目录存在，不存在就自动解压；
- 找到解压目录里的 `x64` 和可选的 `x32`；
- 复制 DXVK DLL 到当前 Prefix 的 `drive_c/windows/system32`；
- 若存在 32 位 DLL，则复制到 `drive_c/windows/syswow64`；
- 在 Prefix 根目录写入 `.dxvk-version` 标记文件。

### 7.2 VKD3D 安装流程

VKD3D 安装时会：

- 确认缓存归档存在，不存在就补下载；
- 确认解压目录存在，不存在就自动解压；
- 递归查找 `x64` 目录；
- 尝试查找 `x86` / `x32` / `i386` 目录；
- 把目标 DLL 复制到 `system32` 和 `syswow64`；
- 在 Prefix 根目录写入 `.vkd3d-version` 标记文件。

### 7.3 这一步为什么重要

因为项目真正运行时依赖的是 Prefix 里的 DLL，而不是缓存目录里的压缩包。

所以：

- 你下载了很多版本，不代表当前游戏就切换了；
- 只有“安装到当前 Prefix”之后，这个游戏才会真正受影响。

## 8. 卸载时，项目会删什么

这里有两个容易混淆的“删除”。

### 8.1 设置页的删除

设置页里的删除，删的是本地缓存：

- DXVK：删除压缩包和已解压目录；
- VKD3D：删除压缩包和已解压目录。

它不会改动任何游戏 Prefix。

### 8.2 游戏设置页的卸载

游戏设置里的卸载，删的是当前 Prefix 中已安装的 DLL 和版本标记：

- DXVK 卸载会移除 `d3d9.dll`、`d3d10core.dll`、`d3d11.dll`、`dxgi.dll` 以及 `.dxvk-version`；
- VKD3D 卸载会移除 `d3d12.dll`、`d3d12core.dll`、`dxil.dll` 以及 `.vkd3d-version`。

因此：

- “删除缓存”不等于“从游戏里卸载”；
- “从游戏里卸载”也不等于“把本地缓存清空”。

## 9. 项目如何判断当前 Prefix 是否真的安装了 DXVK / VKD3D

项目不会只相信 UI 状态，它会再次做实际检测。

### 9.1 DXVK 检测

DXVK 检测逻辑会综合：

- Prefix 中是否存在目标 DLL；
- DLL 是否看起来像 DXVK 二进制；
- Prefix 根目录是否存在 `.dxvk-version`；
- 必要时再做 DLL 内容搜索和文件大小比对。

也就是说，DXVK 版本识别是多级回退的：

- 标记文件
- 二进制内容特征
- 文件大小比对

### 9.2 VKD3D 检测

VKD3D 检测逻辑相对简单：

- 看 `d3d12.dll`、`d3d12core.dll`、`dxil.dll` 是否存在；
- 优先读取 `.vkd3d-version`；
- 如果没有标记文件，再用本地缓存版本做文件大小比对。

### 9.3 这意味着什么

Prefix 配置里的“版本提示”不是最终事实，真正的事实还是 Prefix 里当前存在什么 DLL。

## 10. 启动时 DXVK / VKD3D 会如何影响运行

启动器在真正拉起游戏前，会把 DXVK / VKD3D 视为运行环境的一部分来检查。

项目会记录：

- Prefix 配置中声明的 DXVK / VKD3D 提示值；
- 当前 Prefix 实际检测到的 DXVK / VKD3D 版本；
- 当前找到的 DLL 列表；
- 相关 Proton 图形设置。

并且如果检测到 Prefix 没有安装 DXVK，会给出明确警告：

- 对依赖 DirectX 11/12 的游戏，可能导致黑屏或启动失败；
- 建议回到 `游戏设置 -> 运行环境` 安装。

这也是为什么有些问题明明是“Prefix 里没 DLL”，但用户误以为是下载管理坏了。

## 11. Proton 图形设置会怎样影响 DXVK / VKD3D

图形兼容层不是孤立存在的，它会和 Proton 设置联动。

### 11.1 `dxvk_async`

- 会通过环境变量影响 DXVK 异步着色器编译行为；
- 某些环境下能缓解首轮编译卡顿；
- 也可能引入兼容性差异。

### 11.2 `dxvk_hud`

- 会控制 DXVK HUD 的显示内容；
- 可用来确认当前是否真的走到了 DXVK 路径。

### 11.3 `dxvk_frame_rate`

- 会写入 DXVK 帧率限制相关环境；
- 适合做简单的帧率封顶验证。

### 11.4 `disable_gpu_filter`

- 用于避免项目自动写入 `DXVK_FILTER_DEVICE_NAME`；
- 对多 GPU 环境、独显切换、容器环境排查尤其重要。

### 11.5 `proton_no_d3d12`

- 会设置 `PROTON_NO_D3D12=1`；
- 这等于主动阻断 D3D12 路径；
- 如果你把它打开，VKD3D 的价值会明显下降，甚至直接不走这条路径。

所以 DXVK、VKD3D 和 Proton 图形设置必须一起看，不能拆开理解。

## 12. GPU 选择和 3DMigoto 也会影响图形层结果

项目在启动时还会对图形层做额外注入。

### 12.1 GPU 选择

当你在游戏配置里指定 GPU 时，项目可能会自动设置：

- `DRI_PRIME`
- `__NV_PRIME_RENDER_OFFLOAD`
- `VK_LOADER_DRIVERS_SELECT`
- `DXVK_FILTER_DEVICE_NAME`

这会直接影响 DXVK / VKD3D 最终选到哪块 GPU。

### 12.2 3DMigoto

如果 3DMigoto 启用，项目会倾向于 DX11 路径，并在未显式设置时自动补上：

- `DXVK_ASYNC=1`

所以你在排查图形问题时，不能把 DXVK 和 3DMigoto 当成完全无关的两件事。

## 13. 正确的使用顺序

建议按这个顺序使用：

1. 先确认 Vulkan 驱动本身正常。
2. 在设置页下载需要的 DXVK / VKD3D 版本。
3. 在游戏设置里选好 Wine / Proton。
4. 再把 DXVK / VKD3D 安装到当前游戏 Prefix。
5. 首轮测试先不要叠加 3DMigoto、ShaderFixes、自定义环境变量。
6. 确认能正常启动后，再逐步打开高级功能。

如果你一开始就同时更换 Proton、DXVK、VKD3D、Prefix、3DMigoto 和环境变量，排查价值会非常低。

## 14. 常见误区

- 误区一：我在设置页下载了 DXVK / VKD3D，游戏就会自动用上。不是，还需要到当前游戏的运行环境页安装到 Prefix。
- 误区二：本地缓存里只有压缩包，也算已经准备好了。不完全对，项目可以后续自动解压，但当前还不等于 Prefix 已经有可用 DLL。
- 误区三：切换 DXVK 就一定能解决所有图形问题。不是，Vulkan 驱动、Prefix 污染、GPU 选择、3DMigoto、游戏资源管理都可能是更上游的原因。
- 误区四：VKD3D 只是 DXVK 的附属件。不是，VKD3D 针对的是 D3D12 路径，职责独立。
- 误区五：游戏设置页里显示“已安装”，就一定与缓存版本完全一致。不一定，项目最终还是以 Prefix 中实际 DLL 和标记文件检测为准。

## 15. 什么时候该优先查 DXVK / VKD3D，什么时候不该

更像 DXVK / VKD3D 问题的表现通常是：

- 启动即黑屏；
- 启动即闪退；
- 进入游戏前图形初始化失败；
- HUD、D3D11/D3D12、Vulkan 路径表现异常；
- 切换 Proton 或图形层版本后行为明显改变。

不太像 DXVK / VKD3D 主因的表现通常是：

- 登录页 WebView 超时；
- 网络请求失败；
- Unity `UnloadUnusedAssets / MarkObjects` 长时间卡死；
- 资源流式加载导致的传送卡顿；
- 非图形层的注入器或 SDK 异常。

所以日志分析时，DXVK / VKD3D 要和 `Player.log`、启动日志、Bridge 日志一起看，而不是单独看。

## 16. 推荐的安全工作流

- 至少保留一个稳定可用的 DXVK 版本和一个备用版本。
- VKD3D 只在确实需要 D3D12 路径时启用和测试。
- 切换大版本前，记录当前 Prefix 的可工作组合。
- 出现异常时，先卸载当前 Prefix 里的 DXVK / VKD3D，再重新安装，而不是只删设置页缓存。
- 多 GPU 环境下，先排除 GPU 选择和 `DXVK_FILTER_DEVICE_NAME` 的影响。

## 17. 文档结论

你可以把当前项目中的 DXVK / VKD3D 理解成三层系统：

1. 缓存层：负责从远程获取版本并下载到 `tools/dxvk/` 或 `tools/vkd3d/`。
2. 安装层：负责把缓存中的 DLL 复制到某个具体游戏 Prefix。
3. 运行层：负责在启动前重新检测 DLL、记录版本、注入图形环境变量并给出警告。

所以“DXVK / VKD3D 下载、管理与使用”的真正重点不是下载按钮，而是：

- 你要知道缓存和 Prefix 不是一回事；
- 你要知道 DXVK 和 VKD3D 不是一回事；
- 你要知道项目最终以 Prefix 中的真实 DLL 和启动时环境为准；
- 你要知道很多所谓“图形问题”其实并不一定是 DXVK / VKD3D 的问题。

如果你还没有读过 [游戏下载与主程序配置](02-游戏下载与主程序配置.md) 和 [Proton 下载、管理与使用](03-Proton-下载管理与使用.md)，建议继续阅读。这两篇分别解释启动链、Prefix 和运行环境如何与图形兼容层联动。
