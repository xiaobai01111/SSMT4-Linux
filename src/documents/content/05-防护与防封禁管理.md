# 防护与防封禁管理

最后更新：2026-03-08

本文解释 SSMT4-Linux 里“防护”真正指的是什么，以及它为什么不能被理解成“绝对安全”或“官方认可的防封禁方案”。

如果你理解这条链路，就会知道：

- 为什么项目会在首次使用和下载前要求你做风险确认；
- 为什么有些游戏在启动前必须先通过防护检查，否则会被直接阻止启动；
- 为什么“应用防护”“删除/禁用防护”“切换渠道模式”“安装 Jadeite”“启用沙盒”其实是不同层级的能力；
- 为什么这套系统最多只能降低部分已知风险，而不可能承诺“不会封禁”。

## 1. 先说结论

- 没有任何第三方工具能够保证账号绝对安全，也没有任何第三方工具能够承诺“不会封禁”。
- 本项目中的“防护”本质上是预设驱动的风险控制、运行前校验、文件/配置改写和兼容性补丁，不是官方授权的安全机制。
- 某些功能会降低已知风险，某些功能只是在特定游戏上恢复可启动性，还有一些功能只是让问题更容易被发现。
- 是否启用、何时启用、是否继续使用、是否带着账号进行联机或高敏感操作，最终责任都在用户自身。

## 2. 当前项目里的“防护”其实分成哪几层

这个项目里的“防护”不是一个单一按钮，而是多层系统。

### 2.1 风险确认层

风险确认主要做两件事：

- 首次向导要求用户确认“这是非官方工具，存在反作弊和账号风险”；
- 下载或应用防护前，如果全局还没有确认过风险，会再次要求二次确认。

如果用户没有完成风险确认，启动器会直接阻止启动。

### 2.2 游戏防护层

这是 `下载 / 安装游戏` 或 `下载 / 防护管理` 里看到的那一套“应用安全防护”。

这层当前主要覆盖：

- 遥测服务器屏蔽
- 遥测 DLL 删除
- 渠道参数防护

它是按游戏预设驱动的，不是所有游戏都有。

### 2.3 反作弊兼容层

这一层最典型的是 HoYoverse 游戏的 `Jadeite`。

它不是简单的“风险提示”，而是实打实参与启动链：

- 单独启动时，使用 `jadeite.exe` 包装游戏；
- 与 3DMigoto 同时启用时，Jadeite 会被写入 bridge 配置，由 bridge 负责通过 Jadeite 拉起游戏。

### 2.4 运行时隔离层

这一层是 `游戏设置 -> 系统 -> 沙盒设置` 里的 `bwrap` 沙盒。

它主要作用是运行时隔离，不是反作弊系统，也不是遥测拦截器。

### 2.5 实验功能风险层

像这些功能会显著改变原始启动链：

- 3DMigoto
- Bridge
- Mods / ShaderFixes
- 自定义 DLL
- 自定义环境变量
- 自定义启动命令

它们和“防护”不是同一件事，但会直接影响你的风险水平。

## 3. 这些防护规则从哪里来

项目并不是靠 UI 里手写几段逻辑判断是否要防护，而是从游戏预设里读取。

预设层当前至少提供这些字段：

- `telemetry_servers`
- `telemetry_dlls`
- `channel_protection`
- `require_protection_before_launch`

这意味着：

- 是否支持防护，是预设决定的；
- 需要屏蔽哪些域名、删除哪些 DLL、改哪个渠道字段，也是预设决定的；
- 启动时是“必须已启用，否则阻断”，还是“只提示不强拦”，同样是预设决定的。

所以“防护与防封禁管理”本质上是“预设驱动的风险与兼容策略管理”，不是统一规则硬套到全部游戏上。

## 4. 游戏防护层实际上会做什么

当前项目里的游戏防护主要是三类能力。

### 4.1 遥测服务器屏蔽

项目会从预设读取需要处理的遥测域名，然后检查 `/etc/hosts` 中是否已经存在类似：

- `0.0.0.0 domain.example`

应用防护时：

- 会通过 `pkexec` 获取管理员权限；
- 把对应域名追加到 `/etc/hosts`；
- 会写入带有 `# SSMT4 遥测屏蔽 - {gamePreset}` 标记的条目，便于后续恢复。

恢复防护时：

- 会删除当前项目自己写入的 hosts 条目；
- 也会尝试移除对应的 `0.0.0.0 域名` 记录。

这意味着这部分防护是真正改动系统 hosts 文件的，不是只在内存里做提示。

### 4.2 遥测 DLL 删除

项目还可以按预设定义，从游戏目录中删除特定 DLL。

这里的核心逻辑是：

- 预设给出相对游戏根目录的 DLL 路径；
- 项目检测这些 DLL 是否仍然存在；
- 应用防护时直接删除；
- 启动前检查会再次确认是否还存在未移除项。

所以这部分不是“屏蔽网络”，而是直接改动游戏文件树。

### 4.3 渠道参数防护

这部分比前两者更容易被误解。

项目会读取预设中的渠道配置：

- `config_relative_path`
- `channel_key`
- `init_value`
- `protected_value`
- `default_mode`
- `launch_enforcement`

然后到指定 JSON 配置文件中读取渠道字段的当前值，并判断它是否符合目标模式。

当前支持的模式主要是：

- `init`
- `protected`

一般可以理解成：

- `init`：更适合初始化或特定前置阶段；
- `protected`：更适合联机或正式运行阶段。

但具体含义仍然取决于该游戏预设定义的值，不应该脱离游戏本身硬套理解。

## 5. 渠道模式为什么要单独理解

渠道模式不是“顺手改一下 JSON”那么简单，它有自己的一整套状态管理。

### 5.1 模式会被持久化

项目会把当前选择的渠道模式写入设置库，例如：

- `protection.channel.mode.{gamePreset}`

这意味着：

- 你不是每次打开窗口都回到初始状态；
- 后续再次应用防护时，会优先使用上次保存的模式。

### 5.2 项目会保存原始值

当渠道字段首次从当前值改写到目标值时，项目会把原始值备份到数据库。

这说明它不是“无脑覆盖”，而是有最基本的原值留档。

### 5.3 渠道防护有两种启动约束级别

渠道模式本身还带 `launch_enforcement`：

- `block`
- `warn`

含义是：

- `block`：不满足目标值时，可参与启动阻断；
- `warn`：不满足目标值时，记录为缺失项，但不一定单独构成启动阻断。

因此渠道防护不是单纯的“对”或“错”，而是带有约束等级。

## 6. 启动器到底如何决定“拦不拦”

启动器在真正拉起游戏前，会做一次综合防护检查。

它会汇总：

- 遥测域名是否都已屏蔽；
- 遥测 DLL 是否都已删除；
- 渠道字段是否已经切到目标值；
- 当前游戏预设是否要求“启动前必须完成防护”；
- 当前渠道防护的约束等级是 `block` 还是 `warn`。

最终会生成：

- `supported`
- `enforceAtLaunch`
- `enabled`
- `missing`

如果结论是：

- 当前游戏启动前必须完成防护；
- 但实际状态仍然不满足；

那么启动器会直接阻止启动，并把缺失项写入日志和错误提示。

这就是为什么有些游戏不是“建议应用防护”，而是“不应用就根本不给启动”。

## 7. “防护状态：已启用 / 未启用”在 UI 里是什么意思

`下载 / 安装游戏` 窗口里的防护状态，不是静态标签，而是实时计算结果。

当前 UI 会根据：

- `get_game_protection_info`
- `check_game_protection_status`
- `get_channel_protection_status`

综合显示：

- 该游戏有没有可用防护；
- 当前是否已完成；
- 当前是否属于“未启用将阻止启动”的强制模式；
- 当前渠道值是不是处于预期模式。

所以“已启用”不代表某个单项开关被点亮，而是代表这轮综合检查通过了。

## 8. “应用安全防护”按钮真实会做什么

当前 `应用安全防护` 按钮做的不是单步动作，而是复合动作。

点击后：

1. 先确认该游戏确实有防护项。
2. 弹出确认框，告诉你会应用哪些防护。
3. 调用统一后端入口。
4. 后端依次执行：
   1. 遥测域名屏蔽
   2. 遥测 DLL 删除
   3. 若该游戏配置了渠道防护，则按当前偏好模式改写渠道值
5. 最后重新刷新综合状态。

这意味着“应用安全防护”是一个总入口，不是只做 hosts 屏蔽，也不是只改一个渠道值。

## 9. “删除/禁用防护”当前到底会恢复什么

这是最容易被用户误解的地方。

当前实现里，“删除/禁用防护”主要恢复的是：

- 域名屏蔽
- 文件层防护的可见状态

但渠道模式并不是由这个按钮自动统一恢复的。

后端目前会明确返回提示：

- `restore_telemetry` 不再改写渠道字段；
- 渠道模式由 `set_channel_protection_mode` 单独控制。

所以当前真实边界是：

- “删除/禁用防护”不等于“完全恢复为原始渠道状态”；
- 如果你要改回 `init` 或 `protected`，应该使用专门的“渠道模式”切换按钮。

这也是为什么文档必须把“恢复 hosts / 文件”和“切换渠道值”分开解释。

## 10. Jadeite 属于哪一类能力

`Jadeite` 不是普通的遥测防护项，它更接近“特定游戏的反作弊兼容补丁”。

当前实现中：

- 只对特定 HoYoverse 游戏走这条链；
- 安装入口在 `游戏设置 -> 游戏 -> Jadeite 反作弊补丁`；
- 项目会从默认 API 或用户自定义镜像获取最新版本；
- 下载 zip、解压到 `patch/` 目录，并写入 `.version`。

启动时有两种接法：

- 如果只启用 Jadeite，不启用 3DMigoto，则直接以 `jadeite.exe` 包装游戏启动；
- 如果 Jadeite 和 3DMigoto 同时启用，则会把 Jadeite 路径写入 bridge 配置，让 bridge 在同一启动链里通过 Jadeite 拉起游戏。

因此 Jadeite 不是“提示层”，而是实打实改变了可执行入口和启动链。

## 11. 沙盒和“防封禁”是什么关系

`bwrap` 沙盒是运行时隔离能力，不是账号安全承诺，也不是反作弊绕过器。

它能做的是：

- 限制进程看到的部分文件系统范围；
- 在你开启 `isolate_home` 时提供更严格的 HOME 隔离。

它不能做的是：

- 保证账号不会被识别为异常；
- 替代游戏级防护规则；
- 替代 Jadeite 或遥测处理。

而且当前还有一个实现边界：

- 如果 `umu-run` 已启用，项目会跳过额外的 `bwrap` 沙盒，避免容器嵌套冲突。

所以沙盒只是运行环境隔离项，不能把它等同于“防封禁开关”。

## 12. “防护”和“实验性功能风险”为什么必须分开看

很多人会误以为：

- 我已经应用了防护，所以就可以放心开 3DMigoto、Bridge、Mods、自定义 DLL 和自定义命令。

这是错误理解。

真实情况是：

- 防护系统主要处理的是预设定义的已知风险点；
- 3DMigoto、Bridge、ShaderFixes、Mods、自定义 DLL 和脚本会改变原始启动链、图形链和运行行为；
- 这类变化本身就可能成为风险来源。

所以“防护已启用”不等于“实验性注入层已安全”。

## 13. 正确的使用顺序

建议按这个顺序做：

1. 先完成首次风险确认。
2. 先把游戏下载、主程序路径和运行环境跑通。
3. 打开 `下载 / 防护管理`，确认当前游戏是否支持防护。
4. 若支持，先应用游戏防护，再看状态是否通过。
5. 若有渠道模式，先确认当前处于正确模式。
6. HoYoverse 游戏如需要，再安装 Jadeite。
7. 最后才逐项启用 3DMigoto、Bridge、Mods、ShaderFixes 或自定义环境变量。

如果你一开始就把这些能力全堆上去，出了问题时几乎无法判断究竟是谁导致的。

## 14. 什么时候风险会显著升高

- 你没有完成首次风险确认，却尝试手工绕过启动检查。
- 你在账号登录、支付、实名、联机或敏感场景下测试实验性注入功能。
- 你为了追求 Mod、ShaderFixes 或画面效果，主动关闭必要防护。
- 你在共享 Prefix、脏环境或不明来源的运行目录里继续叠加功能。
- 你把沙盒、Jadeite、3DMigoto、Bridge、自定义环境变量同时打开，却不做分步验证。

## 15. 常见误区

- 误区一：应用了防护，就等于不会封禁。不是，当前实现最多只能降低一部分已知风险和不兼容情况。
- 误区二：防护就是遥测屏蔽。不是，当前还包括 DLL 删除、渠道参数改写、启动前阻断和特定游戏的 Jadeite。
- 误区三：删除/禁用防护会把所有状态都完整恢复。不是，当前域名/文件恢复和渠道模式切换是分开的。
- 误区四：Jadeite 和普通防护是同一层。不是，Jadeite 直接参与启动链，性质更接近反作弊兼容补丁。
- 误区五：启用沙盒就更安全。并不绝对，沙盒主要是运行隔离，兼容性和风险控制不是一回事。

## 16. 什么时候更像是“防护问题”，什么时候不是

更像防护问题的表现通常是：

- 启动前直接提示“未启用应用防护，已阻止启动”；
- 下载/防护管理页里显示“未启用（将阻止启动）”；
- 渠道值不正确，且当前模式与目标值不一致；
- HoYoverse 游戏缺少 Jadeite，导致根本无法正常拉起；
- 日志中明确出现 `protection check failed`、渠道模式不匹配或缺少防护项。

不太像防护主因的表现通常是：

- Unity `Player.log` 中的资源卸载长卡顿；
- DXVK / VKD3D 图形初始化或显卡驱动问题；
- 3DMigoto 注入失败；
- WebView 超时；
- 普通网络波动、下载源错误或文件损坏。

所以排查时不要把所有问题都归到“防护没开”。

## 17. 推荐的安全工作流

- 对需要防护的游戏，先确认 `下载 / 防护管理` 页面是否显示可用防护项。
- 应用防护后，再看综合状态是否真正变成“已启用”。
- 如果有渠道模式，明确记录自己当前处于 `init` 还是 `protected`。
- 对 HoYoverse 游戏，把 Jadeite 是否安装、版本号和 patch 目录一起记录。
- 每次只改一个变量，避免把“防护问题”和“运行环境问题”混在一起。
- 如果你必须关闭某项防护，先保留日志和配置快照，再做恢复操作。

## 18. 文档结论

你可以把当前项目里的“防护与防封禁管理”理解成四层系统：

1. 风险确认层：没有完成风险确认，直接不给启动。
2. 游戏防护层：基于预设处理遥测域名、遥测 DLL 和渠道参数。
3. 反作弊兼容层：以 Jadeite 为代表，直接参与特定游戏的启动链。
4. 运行隔离层：以 bwrap 沙盒为代表，提供运行期隔离，但不承诺账号安全。

所以“防护”真正解决的是：

- 已知风险点的规避；
- 已知配置缺失的启动前拦截；
- 部分游戏在 Linux 上的可启动性修复；

而它不能解决的是：

- 一切反作弊风险；
- 一切账号风控；
- 一切实验性注入导致的问题；
- 一切图形、性能、网络和游戏逻辑问题。

如果你还没有读过 [《服务条款》与风险声明](00-服务条款与风险声明.md)、[项目风险与要求](01-项目风险与要求.md) 和 [游戏下载与主程序配置](02-游戏下载与主程序配置.md)，建议继续阅读。这三篇分别解释法律边界、工程风险边界以及下载/防护入口如何接入实际启动流程。
