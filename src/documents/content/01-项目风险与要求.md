# 项目风险与要求

最后更新：2026-03-08

本文是对 [《服务条款》与风险声明](00-服务条款与风险声明.md) 的工程化补充。

它不是法律文本的重复版，而是面向实际使用者的操作型文档，重点回答四个问题：

1. 你需要具备什么前提，才适合使用本项目。
2. 你面对的风险到底来自哪里，而不是笼统地说“有风险”。
3. 哪些功能属于高风险或极高风险，为什么。
4. 你在启用、排查、反馈和长期使用时，应遵守什么边界。

## 1. 文档定位

- SSMT4-Linux 是一套非官方 Linux 游戏启动与管理工具链，不是官方启动器的替代保证方案。
- 本文强调的是“工程现实”和“风险控制”，不是营销承诺。
- 如果你需要的是零风险、零维护、官方背书、官方兼容承诺或官方售后支持，那么本项目不适合你。

## 2. 适用对象

本项目更适合以下用户：

- 能自行管理 Linux 文件系统、用户目录、权限和可执行文件路径的用户。
- 能理解 Wine、Proton、DXVK、Prefix、兼容层并非官方运行环境的用户。
- 能接受“先看日志、再定位问题、最后调参数”这一工作方式的用户。
- 愿意为自己的环境、账号、数据和实验性配置负责的用户。

本项目不适合以下用户：

- 无法接受账号处罚、功能限制或运行异常风险的用户。
- 希望通过一键操作长期稳定解决所有兼容性问题的用户。
- 不愿保留日志、不愿备份、不愿描述复现步骤、只接受立即修好的用户。
- 打算将注入、Hook、图形替换、脚本能力用于不当用途的用户。

## 3. 使用前最低要求

在开始之前，你至少应满足以下条件：

- 知道当前系统的 Linux 发行版、桌面环境、显卡型号和驱动基本情况。
- 能分辨游戏启动器、本体主程序、更新器、辅助工具等不同可执行文件。
- 知道当前游戏使用的 Proton/Wine 版本、Prefix 路径和 DXVK 状态。
- 知道如何查看 `Player.log`、启动日志、Bridge 输出、图形层日志等基础材料。
- 明白“能启动”不等于“稳定可用”，一次可运行不代表长期可运行。

如果你连当前游戏实际用的是哪个 `.exe`、哪个 Prefix、哪个 Proton 都不清楚，那么先不要启用高风险功能。

## 4. 风险不是抽象概念，而是分层存在

本项目的风险主要来自以下几层：

### 4.1 运行环境层风险

- Wine/Proton 会改变 Windows 游戏在 Linux 下的系统调用、文件访问、注册表、图形接口和线程行为。
- 同一款游戏在不同 Proton 版本、不同驱动、不同显卡、不同桌面环境下，表现可能完全不同。
- 某次更新可能修好一个问题，也可能引入新的回归。

### 4.2 图形与性能层风险

- DXVK、VKD3D、图形驱动、Vulkan 实现以及额外图形中间层会共同影响黑屏、闪退、卡顿、掉帧、场景加载时间和显存行为。
- 某些问题看起来像“游戏坏了”，实际可能来自图形层兼容性、着色器编译、资源卸载、显存探测异常或驱动行为差异。
- 启用 ShaderFixes、纹理替换或其他图形改写后，渲染链复杂度会明显上升。

### 4.3 启动链与注入层风险

- Bridge、3DMigoto、DLL 注入、脚本钩子、自定义启动命令和额外库加载，都会改变原始启动链。
- 一旦启动链发生改变，故障点就不再只属于游戏本身，还可能来自桥接程序、导入器、动态库加载顺序、路径劫持或外部配置。
- 这类风险往往不是“性能变差一点”，而是直接导致进程挂起、窗口无响应、图形错误或无法进入游戏。

### 4.4 账号与平台层风险

- 非官方兼容环境、异常注入、图形替换、运行链改写、系统行为偏差，都可能被平台、游戏或反作弊服务视为异常。
- 这类风险未必会立即表现为封禁，也可能表现为告警、功能受限、风控记录、验证增强或后续限制。
- 没有人能够为“绝对不会触发平台风险”做保证。

### 4.5 数据与系统层风险

- 配置错误可能导致游戏目录、Prefix、缓存、存档、下载数据和 Mod 目录被污染或覆盖。
- 高负载问题还可能拖慢桌面环境、占满磁盘、拉高显存/内存占用，甚至影响系统稳定性。
- 来源不明的补丁、脚本、DLL 或 Mod 还可能引入安全风险。

## 5. 风险等级建议

为了方便实际判断，可以把常见操作按风险等级理解为：

### 5.1 低风险

- 仅使用 Home 页面启动已经正确配置的游戏。
- 查看日志、浏览文档、管理基础设置。
- 在不改动实验性开关的前提下切换当前游戏。

低风险不等于无风险，只表示它通常不主动改变运行链。

### 5.2 中风险

- 修改主程序路径。
- 切换 Proton/Wine 版本。
- 安装或切换 DXVK、VKD3D。
- 调整 Prefix、环境变量、系统兼容项。

这类操作通常会改变运行环境，但不一定直接引入进程注入。

### 5.3 高风险

- 启用 3DMigoto。
- 使用 ShaderFixes、Mods、图形替换。
- 修改 Bridge 配置或导入器目录。
- 使用自定义启动命令、外部脚本、额外 DLL。

这类操作已经开始改变渲染链、启动链或资源加载行为。

### 5.4 极高风险

- 同时叠加多个实验性组件，并在未保留基线配置的情况下长期使用。
- 在账号登录、支付、验证、强联网对抗或高敏感业务流程中测试注入层。
- 使用来源不明的补丁、Mod、DLL、脚本或二进制替换文件。
- 在不理解风险来源的前提下，盲目尝试“别人能用所以我也全开”。

极高风险配置下，出现账号、数据、稳定性问题的概率会显著上升。

## 6. 必须接受的现实边界

在继续使用本项目之前，你应明确接受以下事实：

- 不是所有游戏都适合在 Linux/Wine/Proton 下长期稳定运行。
- 不是所有问题都能通过更换一个版本号解决。
- 不是所有问题都发生在“游戏代码”本身，很多问题来自兼容层、运行链和系统环境。
- 开发者可以帮助你缩小问题范围，但不会为用户自己的高风险配置背书。
- 项目会尽量提供开关、日志、提示和文档，但不承诺所有问题都能修复。

## 7. 启用实验性功能前的硬性要求

在启用 3DMigoto、Bridge、Mods、ShaderFixes、脚本钩子、自定义 DLL 或类似功能前，至少应做到：

1. 备份游戏目录、Prefix、配置文件、重要存档和 Mod 目录。
2. 记录当前可工作的配置基线。
3. 明确当前准备修改的是哪一层：启动链、渲染链、路径层还是资源层。
4. 一次只改一个变量，避免多个高风险项同时变动。
5. 保证日志可留存，且知道异常发生后去哪里看。

如果你做不到以上 5 点，就不应该启用实验性功能。

## 8. 明确不建议的做法

以下做法会显著放大风险或降低排查效率：

- 在没有日志的情况下频繁切换 Proton、DXVK、VKD3D、Prefix 和注入开关。
- 把来源不明的 Mod、补丁、脚本、DLL 直接放进游戏目录或 Mod 目录。
- 在一次测试中同时改动 3DMigoto、Bridge、DXVK、主程序路径和环境变量。
- 遇到问题后先删文件、覆盖目录、重建 Prefix，却不保留任何现场信息。
- 在登录、支付、实名、网页验证或联机敏感阶段测试注入层和图形替换功能。
- 把别人的配置直接照搬到完全不同的游戏版本、显卡环境或 Linux 发行版上。

## 9. 推荐的安全工作流

为了降低问题复杂度，建议按以下顺序使用项目：

1. 先完成初始化和基础设置。
2. 先确认主程序路径正确。
3. 先让游戏在“无实验性功能”的情况下可以启动。
4. 再确认 Proton、Prefix、DXVK 等基础环境稳定。
5. 最后才逐项开启 3DMigoto、Mods、Bridge 或其他实验性功能。

如果你跳过前 4 步直接上实验性层，那么你很难区分问题到底来自游戏、环境还是注入链。

## 10. 出现问题时的标准处理顺序

发生异常后，建议遵循以下顺序：

1. 先复现问题并记录发生时间点。
2. 保留日志，不要立即覆盖现场。
3. 确认最近改动了哪些配置。
4. 优先关闭最近新增的实验性功能。
5. 恢复到最近一个明确可工作的配置。
6. 再逐项重新开启，确认真正的触发项。

不要把“复现问题”和“清空环境”放在同一步里做。

## 11. 反馈问题时你应提供什么

高质量反馈至少应包含：

- 游戏名称、区服、版本信息。
- Linux 发行版、桌面环境、显卡与驱动。
- 使用的 Proton/Wine 版本、DXVK 状态、Prefix 路径。
- 是否启用了 3DMigoto、Bridge、Mods、ShaderFixes、自定义 DLL 或自定义命令。
- 复现步骤、实际现象、是否稳定复现。
- 对应日志路径、关键报错、问题发生时间点。

如果这些信息缺失，问题定位通常会大幅变慢，甚至无法判断责任层级。

## 12. 项目支持边界

项目可以尽量提供以下帮助：

- 帮你判断问题更像发生在游戏、兼容层、图形层还是注入层。
- 帮你阅读日志、缩小排查范围、识别明显配置错误。
- 帮你整理更合理的测试顺序和配置基线。

项目不承诺以下结果：

- 保证某个游戏一定稳定运行。
- 保证任何实验性功能都可长期使用。
- 为第三方组件、社区 Mod、外部 DLL 或来源不明脚本承担责任。
- 为账号风险、平台处罚或数据损坏提供担保。

## 13. 何时你应该立即停止继续测试

出现以下任一情况时，建议立刻停止继续叠加测试：

- 出现账号告警、验证异常、平台限制或其他明显风控迹象。
- 游戏目录、Prefix、存档或配置已经出现异常覆盖、损坏或混乱。
- 系统开始出现明显的资源耗尽、桌面卡死、进程堆积或长时间无响应。
- 你已经不清楚当前环境到底改过哪些内容。
- 你无法判断某个补丁、Mod、脚本或 DLL 的来源和用途。

停止测试不是失败，而是为了避免问题从“可排查”升级为“不可恢复”。

## 14. 最终确认

继续使用 SSMT4-Linux，尤其是在启用实验性功能前，你应确认：

1. 你已经阅读 [《服务条款》与风险声明](00-服务条款与风险声明.md)。
2. 你理解本项目的风险来自运行环境、图形层、注入层、平台层和数据层，而不是单一来源。
3. 你理解高风险功能不是默认能力，而是需要谨慎开启的实验性工具。
4. 你愿意对自己的配置、数据、账号和测试行为负责。
5. 你接受项目提供的是工具、文档和排查能力，而不是官方保证。
