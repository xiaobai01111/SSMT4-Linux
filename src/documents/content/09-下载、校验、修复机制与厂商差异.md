# 下载、校验、修复机制与厂商差异

最后更新：2026-03-08

这篇文档解释 SSMT4-Linux 中“下载 / 更新 / 校验 / 修复”这整条链路的真实工作方式。

它不是“怎么点按钮”的教程，而是用来回答这些问题：

- 为什么有些游戏支持自动下载，有些只能下载官方启动器安装器；
- 为什么同一个游戏换了服务器后，原本存在的本地版本会被当成“未安装”；
- 为什么有些游戏支持“修复异常文件”，有些只能整包重下；
- 为什么下载完成后 UI 还会继续进入“校验中”；
- 为什么关闭下载窗口后任务不会停。

## 1. 先理解三层结构

当前项目的下载链路可以分成三层。

### 1.1 配置层

这一层负责决定：

- 当前游戏是否支持自动下载；
- 使用哪个 `launcherApi`；
- 是否需要 `biz_prefix`；
- 默认安装目录是什么；
- 下载模式是 `full_game` 还是 `launcher_installer`；
- 音频语言列表和服务器列表来自哪里。

这些信息既可能来自游戏预设，也可能被当前游戏的 `Config.json` 覆盖。

### 1.2 调度层

这一层主要在 `game_downloader` 命令和前端 `downloadStore` 中。

它负责：

- 判断当前状态是 `NeedInstall`、`NeedUpdate`、`StartGame` 还是 `NetworkError`；
- 决定本轮是“下载”“更新”“校验”“修复”还是“下载官方启动器安装器”；
- 统一维护当前任务、进度、暂停/恢复和异常文件列表；
- 在任务成功后更新本地版本和下载源元数据。

### 1.3 厂商实现层

这一层才是真正的下载器实现。

当前主要有三条分支：

- Kuro Games
- HoYoverse
- Snowbreak

它们的清单结构、增量逻辑、校验来源和修复能力都不一样。

## 2. 下载窗口里的模式不是一个名字游戏

当前下载窗口有两种模式。

### 2.1 `full_game`

这是“真正的游戏资源下载 / 更新 / 校验 / 修复”模式。

在这个模式下，主操作围绕这些动作：

- 下载完整游戏
- 更新游戏
- 校验文件
- 修复异常文件

### 2.2 `launcher_installer`

这是“只下载官方启动器安装器”的模式。

这类模式的特点是：

- `launcherApi` 返回的不是完整游戏资源清单；
- 它返回的是 `version` + `exe_url` 这类安装器信息；
- 下载完成后，项目会把 `gamePath` 自动指向下载到的安装器；
- 后续你仍可能需要用官方安装器把游戏真正装好，再回到项目里把主程序路径改成实际游戏 exe。

所以：

- `full_game` 是面向游戏资源本体；
- `launcher_installer` 是面向厂商启动器安装包。

## 3. 状态是怎么判断出来的

下载窗口里显示“需要安装”“需要更新”“已是最新版本”，不是简单看目录里有没有文件，而是结合了远端状态、本地版本和下载源范围。

## 3.1 本地版本不是只看一个文件

当前项目会按一定顺序探测本地版本。

常见来源包括：

- `.version`
- `launcherDownloadConfig.json`

并且不仅看当前目录，还会向上探测若干级父目录。

这意味着：

- 同一个游戏的目录结构稍有不同，项目也可能仍然认出本地版本；
- 但如果结构偏差太大，仍然可能被当成“未安装”。

## 3.2 本地版本还会按下载源做作用域限制

这是非常关键的一点。

项目不仅记录本地版本，还会记录下载来源元数据：

- `launcher_api`
- 对 HoYoverse 来说还包括 `biz_prefix`

这些信息会写入：

- `.download_source_meta.json`

后续判断状态时，如果当前选择的服务器或 `biz_prefix` 和保存时不一致，项目会把本地版本视为“不属于当前下载源”，从而返回“需要安装”或“不认这个版本”。

所以你会看到一种常见现象：

- 目录里明明已经有资源；
- 但切换服务器后，状态却像是没装一样。

这不是纯粹的 bug，而是当前实现刻意做的“下载源隔离”。

## 3.3 状态机本身很简单

不论具体厂商，核心状态大体只有四类：

- `NeedInstall`
- `NeedUpdate`
- `StartGame`
- `NetworkError`

区别在于每个厂商如何得到：

- 远端版本
- 本地版本
- 是否支持增量

## 4. 任务不是跟着窗口活的

前端下载体系目前是全局 store 驱动，而不是下载弹窗局部状态。

这意味着：

- 下载、校验进度监听是全局注册的；
- 关闭下载窗口不会中断活动任务；
- 只要任务还在，重新打开窗口仍能看到当前进度。

这也是为什么当前设计里：

- 下载弹窗是控制台；
- 任务本体并不绑定在弹窗生命周期上。

## 5. 下载完成后为什么还会继续“校验中”

因为当前主流程在“下载 / 更新”成功后，前端会自动追加一次校验。

这一点很重要：

- 下载完成不代表文件一定完全可信；
- 自动校验会把异常文件列表保存下来；
- 如果校验失败，UI 会进入“下载完成但自动校验失败”的提示，而不是直接静默成功。

对于支持单文件修复的分支，后续“修复异常文件”按钮就是基于这份异常列表。

## 6. Kuro Games 分支

这一支最接近“通用资源索引下载器”。

### 6.1 远端数据结构

Kuro 分支会先读取 launcher API，然后从里面找到：

- `default`
- `config`
- `cdnList`
- `indexFile`
- `patchConfig`

如果 launcher API 本身其实是“启动器安装器 payload”，而不是完整游戏下载 payload，当前解析会直接报错并提示应使用 `launcher_installer` 模式。

### 6.2 全量下载

Kuro 的全量下载特点是：

- 先读取资源索引；
- 对每个资源校验 `sha256` 或 `md5`；
- 已存在且校验通过的文件会被跳过；
- 未通过的文件进入下载队列；
- 下载使用高并发模型。

当前实现的设计重点是：

- 下载前先做预检查，不重复下载已经正确的文件；
- 并行度较高，适合资源条目多的小文件场景；
- 单个文件支持断点续传和下载后完整性校验。

### 6.3 更新

Kuro 当前前端主流程走的是“全量比对更新”。

也就是说：

- 它会对当前资源树逐文件比对；
- 校验通过的跳过；
- 校验失败或缺失的重新下载。

后端其实还提供了单独的增量补丁更新命令：

- `update_game_patch`

但当前主界面主流程并不直接以它作为默认入口。

### 6.4 增量补丁能力

Kuro 的补丁能力依赖：

- `patchConfig`
- 补丁索引
- `hpatchz`

当前实现会：

- 下载补丁索引对应的差分文件；
- 必要时自动准备 `hpatchz`；
- 应用 `.krdiff` 或类似补丁；
- 写回最新版本。

所以“支持增量”这件事，在 Kuro 分支里不是抽象概念，而是真有专门补丁路径。

### 6.5 校验

Kuro 的校验是当前三家里最完整的一支。

它有两个重要特点：

第一，校验分阶段：

- 第一阶段先做文件存在与大小筛选；
- 第二阶段再做哈希校验。

第二，校验用了本地 MD5 缓存：

- 以路径、大小、mtime 做缓存键；
- 避免每次都重新计算所有文件 MD5。

这意味着：

- 大型目录的重复校验成本会明显下降；
- 但如果文件被外部改动，mtime 变化后缓存也会自然失效。

### 6.6 修复

Kuro 当前支持按异常列表做单文件修复。

这一能力依赖：

- 前一次校验留下的异常文件列表；
- 当前资源索引仍能定位到这些条目；
- 下载器重新拉取对应文件。

这就是为什么 UI 里“修复异常文件”只对部分游戏可用。

## 7. HoYoverse 分支

这一支和 Kuro 差异很大，核心关键词是：

- `biz_prefix`
- 分包
- 资源清单回退
- 不支持单文件修复

### 7.1 `biz_prefix` 是硬前提

HoYoverse 分支不能只靠 `launcherApi`。

它还需要：

- `biz_prefix`

这是为了在同类 launcher API 里精确找到目标游戏包。

如果缺少它，当前实现会直接拒绝下载相关操作。

所以：

- 服务器切换不只是换 URL；
- 对 HoYoverse 来说，本质上还在换“当前业务包范围”。

### 7.2 下载的是游戏包，不是简单文件索引

HoYoverse 分支先会拉取游戏包描述，然后得到：

- 主包
- 语言包
- 可用补丁
- 资源清单链接

实际下载时，不是像 Kuro 那样直接遍历一个统一资源索引，而是走安装包与安装计划。

### 7.3 更新和安装会有安装阶段

HoYoverse 下载流程不只是“文件落盘”。

它往往还包括：

- 安装包缓存
- 解压安装
- 可能的 `hdiff` 补丁应用
- `deletefiles.txt` 清理旧文件
- 最终写回版本文件

所以在 UI 里你会看到：

- `download`
- `install`

两个不同阶段，而不是单一的“下载中”。

### 7.4 本地版本写回更完整

HoYoverse 分支安装完成后，会同时写：

- `.version`
- `launcherDownloadConfig.json`

这样做是为了兼容不同版本探测路径。

### 7.5 校验不是只有一种来源

HoYoverse 的校验逻辑有明确的回退层次。

优先级大致是：

1. 官方资源清单
2. 本地 `pkg_version` 清单
3. 安装包缓存

如果前一种拿不到，才回退到后一种。

这说明 HoYoverse 校验更依赖“现有安装语义”和“包管理痕迹”，而不是纯文件系统资源索引。

### 7.6 HoYoverse 当前不支持按异常列表单文件修复

这是当前实现里必须明确写出来的边界。

现在的行为是：

- 可以校验；
- 可以得到失败列表；
- 但不支持像 Kuro 那样按失败条目逐个重拉文件。

原因不是 UI 没写，而是底层安装模型不同：

- 文件可能来自压缩分包；
- 有时需要从整包解压或重新执行安装步骤；
- 单文件重下并不等价于真正修复完成。

所以当前文案会明确提示：

- HoYoverse 目前请使用重新下载，而不是单文件修复。

### 7.7 官方启动器安装器模式

某些 HoYoverse 或同类厂商下载源，还会返回“启动器安装器 payload”。

这时候当前项目会走：

- `launcher_installer` 模式

它会：

- 读取远端 `version` 和 `exe_url`
- 下载安装器 exe
- 写入 `.launcher_installer_meta.json`
- 把 `gamePath` 自动指向安装器

所以你看到“下载完成后主程序路径变成了 LauncherInstaller.exe”，这是设计行为，不是误改。

## 8. Snowbreak 分支

Snowbreak 的实现思路又和前两家不同。

### 8.1 核心是 manifest + CDN 轮询

Snowbreak 不走 Kuro 那种统一 launcher default/config 结构，也不走 HoYoverse 那种游戏包模式。

它依赖：

- manifest
- CDN base URL 列表
- 源策略

### 8.2 有源策略

当前实现支持：

- `official_first`
- `community_first`

这会影响 manifest 和 CDN 的选择顺序。

所以 Snowbreak 的“下载源选择”不仅是一个 URL，而是一套优先级策略。

### 8.3 下载和更新共用同一条主流程

Snowbreak 当前没有单独拆出一条“真正不同的更新路径”。

下载与更新都走：

- 拉远端 manifest
- 读取本地 manifest
- 对比差异
- 只下载缺失或哈希不匹配文件

所以在 Snowbreak 这里，“下载”和“更新”本质上更接近同一个动作。

### 8.4 校验会尝试重下异常文件

Snowbreak 校验时，如果发现文件缺失或 MD5 不匹配，会尝试立刻从 CDN 重新下载。

这点和 HoYoverse 很不一样。

因此它的校验结果里可能出现：

- `verified_ok`
- `redownloaded`

也就是说它的“校验”本身就带一定的自动修补意味。

### 8.5 但仍不支持前端的按列表单文件修复

虽然 Snowbreak 的校验会自动重下异常文件，但当前前端仍不会开放“修复异常文件”按钮。

后端也明确把这条能力标成：

- 暂不支持按异常列表单文件修复

所以不要把“校验会自动补下”误解成“已经有完整的 repair 模式”。

## 9. 下载源与版本元数据为什么很重要

下载流程成功后，当前实现通常会写回两类信息：

- 本地版本
- 下载源元数据

这一步的价值在于：

- 下次打开窗口时能判断当前目录属于哪个服务器 / 哪个业务范围；
- 避免把不同服务器或不同业务包的资源混认成同一套版本。

但它也带来一个实际影响：

- 同目录切换服务器后，状态可能变成 `NeedInstall`；
- 不是因为文件瞬间消失了；
- 而是因为当前逻辑不再承认它属于新的下载源。

## 10. 校验成功后为什么有时会更新本地版本，有时不会

当前项目只有在“结果足够干净”时，才会写回本地版本。

例如 Kuro 分支中：

- 校验通过且没有失败文件，才会写回最新版本；
- 修复完成且没有残余失败文件，才会写回最新版本；
- 如果仍有失败文件，就不会把版本号强行提升。

这个策略的目的很明确：

- 避免“版本号已经显示最新，但文件实际仍然坏着”。

## 11. 任务并发与保护边界

当前下载相关操作在开始前都会尝试获取写锁。

这意味着：

- 同一个游戏目录不会被多个下载 / 校验 / 修复任务同时乱写；
- 某些情况下切区服或同目录并发操作会被阻止。

这不是多余限制，而是为了避免资源目录和版本元数据互相覆盖。

## 12. 常见误区

- 误区一：下载完成就一定已经完全可启动。不是，当前主流程往往会继续自动校验。
- 误区二：本地目录里有文件，状态就一定不是“需要安装”。不是，下载源元数据不匹配时会被视为不属于当前服务器。
- 误区三：所有游戏都支持“修复异常文件”。不是，当前基本只有 Kuro 分支支持完整的按列表修复。
- 误区四：HoYoverse 的校验失败后应该自动重下异常文件。不是，当前实现明确没有这条完整能力。
- 误区五：Snowbreak 没有 repair 按钮就完全不能修。不是，它的校验本身会尝试重下异常文件，但没有独立公开的列表修复流程。
- 误区六：关闭下载窗口会中断任务。不是，任务由全局下载 store 持有。

## 13. 推荐工作流

### 13.1 首次安装

1. 先确认当前游戏的 `launcherApi`、服务器和安装目录。
2. 明确当前是 `full_game` 还是 `launcher_installer`。
3. 开始下载后不要频繁切换服务器。
4. 下载完成后等自动校验跑完，再判断是否成功。
5. 如果是启动器安装器模式，记得后续把主程序路径改成真正游戏 exe。

### 13.2 更新

1. 先确认当前游戏目录和当前服务器是否匹配。
2. 如果切过区服或服务器，先接受状态判断可能回到“需要安装”。
3. Kuro 可优先走正常更新，必要时再看补丁路径。
4. HoYoverse 更新失败时，不要预期单文件修复能兜底。

### 13.3 排障

1. 先看状态判断是否合理。
2. 再看是否是下载源元数据不匹配。
3. 再区分是下载失败、安装失败、校验失败还是修复失败。
4. 最后再按厂商分支去看更深层日志。
