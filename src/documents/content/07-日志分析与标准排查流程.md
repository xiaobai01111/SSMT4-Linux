# 日志分析与标准排查流程

最后更新：2026-03-08

这篇文档不是“列日志文件名”的索引，而是一套面向 SSMT4-Linux 当前实现的排障流程。

目标只有三个：

- 先判断问题位于哪一层。
- 再决定应该优先看哪一类日志。
- 最后用最少的改动缩小范围，而不是一上来同时改一堆设置。

## 1. 先建立正确的排障思路

排障时最常见的错误，不是不会看日志，而是先把现场破坏了。

典型错误包括：

- 问题刚出现就连续切 Proton、Prefix、DXVK、VKD3D、3DMigoto、Bridge、环境变量。
- 没记录时间点，导致后面根本不知道哪一段日志对应哪一次测试。
- 只截一张“无响应”图片，不保留 `Player.log`、启动日志、Bridge 输出。
- 把每一条 warning 都当成根因，忽略真正重复出现、能和时间点对齐的异常。

标准思路应该是：

1. 固定当前配置，不要立刻大改。
2. 记录问题发生时间、操作步骤、当前启用的关键开关。
3. 先看“启动链”是否通，再看“游戏本体”是否卡，再看“图形层”或“实验性功能”是否叠加放大问题。
4. 每次只改一层，再复测。

## 2. 出现问题后的第一步操作

问题刚出现时，优先做这些动作：

1. 在 Home 打开“游戏日志窗口”，然后重新复现一次。
2. 记下准确时间点，至少精确到分钟，最好能精确到秒。
3. 记下当前选中的游戏、使用的 Proton/Wine、Prefix、DXVK、VKD3D 状态。
4. 记下是否启用了 3DMigoto、Bridge、Mods、ShaderFixes、Jadeite、自定义环境变量或自定义启动参数。
5. 不要立刻覆盖 `Player.log`、`Player-prev.log`、`bridge-output.log` 等现场材料。

如果你没有先开“游戏日志窗口”，也不是完全没法查，但会损失当前会话的启动链细节。

## 3. 当前项目里的日志分层

SSMT4-Linux 的日志不是一份文件，而是多层组合。

### 3.1 应用运行日志

这是 SSMT4-Linux 自己的运行日志，对应“设置 -> 日志查看器”。

它的特点：

- 当前实现默认不再写持久化 `ssmt4.log` 文件。
- 运行日志主要保存在当前会话内存里。
- 适合看界面操作、命令调用、启动器状态、路径解析、错误提示。
- 适合定位“为什么按钮没生效”“为什么启动命令没组出来”“为什么某个命令直接报错”。

这类日志不是长期历史档案。你关闭应用、重开新会话后，就不应该指望它替代历史现场。

### 3.2 游戏日志窗口

这是 Home 里的“游戏日志窗口”，按当前游戏单独维护一份会话缓冲。

它的特点：

- 适合看启动链条、外部进程 stdout/stderr、环境变量快照、运行器选择、部分 Bridge 回读结果。
- 只覆盖当前打开窗口后的会话，不等于永久历史日志。
- 如果你想分析“点击启动后到底发生了什么”，它通常是第一优先级材料。

如果你要反馈“点启动后没反应”“刚启动就退出”“启动器卡住”，请优先保留这份内容。

### 3.3 `Player.log` 与 `Player-prev.log`

这是 Unity 游戏本体最关键的日志层。

适合看这些问题：

- 进入游戏后掉帧、卡顿、无响应、长时间加载
- 场景切换、传送、资源卸载、对象扫描
- Unity 异常、插件异常、部分脚本栈

排查时要注意：

- `Player.log` 反映当前或最近一次运行。
- `Player-prev.log` 经常能补足“上一轮到底怎么结束的”。
- 对于 Unity 游戏，很多“无响应”根因其实不在启动器，而在 `Player.log` 的主线程长卡。

### 3.4 `launcher.log` 与 `games.log`

这类日志通常属于厂商启动器或游戏外层调度程序。

适合看这些问题：

- 启动器有没有真正拉起游戏进程
- 启动链停在启动器、修复器、更新器还是进入了游戏本体
- 外层配置、驱动识别、修复入口、版本判断

如果你看到桌面上只有启动器界面，游戏本体根本没起来，那么优先看这层。

### 3.5 `sdklogs`

这类日志一般位于游戏数据目录或 Prefix 中的 SDK 日志目录，常见内容包括：

- 登录状态
- WebView
- 公告或活动页加载
- 支付、商品、通知
- 网络诊断

它更适合解释这些问题：

- 公告页、登录页、网页组件加载异常
- SDK 上报失败
- WebSocket、活动页、网络探测错误

它通常不是“地图里掉帧”和“传送长卡”的第一根因。

### 3.6 `CrashSightLog`

如果游戏或 SDK 接了 CrashSight，这类日志常用于记录异常上报时点。

它的价值在于：

- 能说明异常是不是反复发生；
- 能帮助你把某个时间点和 `Player.log` 或启动链日志对齐；
- 但它本身往往不是最完整的根因材料。

### 3.7 Bridge 日志与配置

如果启用了 Bridge，至少要同时看两类文件：

- `Cache/bridge/bridge-output.log`
- `Cache/bridge/bridge-config.json`

它们的作用不同：

- `bridge-output.log` 用于看注入过程、桥接行为、外部输出。
- `bridge-config.json` 用于确认这次启动到底生成了什么配置，特别是 importer、进程匹配、延迟和注入策略。

当前实现还会在游戏子进程退出后，把 `bridge-output.log` 回读到“游戏日志窗口”里。所以你经常能在一个地方先看到桥接摘要，再决定要不要去原始文件做细查。

### 3.8 3DMigoto 的 `d3d11_log.txt`

如果你要排查 3DMigoto、ShaderFixes、纹理替换、draw call 相关问题，就需要这类日志。

但要注意一个前提：

- 没有 `d3d11_log.txt`，不一定代表 3DMigoto 没加载。
- 很多时候只是配置里没有开启对应日志项。

所以排查这层问题时，不能简单地以“有没有日志文件”直接下结论。

## 4. 先按现象分流，再决定看哪份日志

### 4.1 点击启动后没反应，或者瞬间退出

优先顺序：

1. 游戏日志窗口
2. 应用运行日志
3. `launcher.log` / `games.log`
4. 再确认主程序路径和启动链

这类问题先不要急着翻 `Player.log`，因为很多时候游戏本体压根没起来。

重点看：

- 选择的可执行文件是不是正确
- 运行器最终是 `umu-run`、直接 Proton、Wine，还是 Pressure Vessel
- 工作目录、Prefix、环境变量是否明显异常
- 启动器是否真正创建了目标进程

### 4.2 启动器能开，但进不去游戏本体

优先顺序：

1. `launcher.log` / `games.log`
2. 游戏日志窗口
3. 防护相关日志与状态
4. 必要时再看 SDK 日志

重点看：

- 外层启动器是否卡在更新、修复、参数校验或主程序调度
- 是否被防护检查、渠道检查、路径检查拦住
- 是否进入了厂商启动链但没有进入 Unity/UE 本体

### 4.3 进入游戏后无响应、掉帧、系统卡、传送加载久

优先顺序：

1. `Player.log`
2. `Player-prev.log`
3. 游戏日志窗口
4. Bridge / 3DMigoto / 图形层日志

这类问题很多时候是“游戏本体正在长时间阻塞”，而不是“启动器出错”。

如果你看到这些特征，要高度怀疑 Unity 主线程卡顿：

- `Unloading unused Assets`
- `MarkObjects`
- 大量资源卸载
- 场景切换或传送时明显的秒级停顿

### 4.4 启用 3DMigoto、Mods、ShaderFixes、Bridge 后才出问题

优先顺序：

1. `bridge-config.json`
2. `bridge-output.log`
3. `d3d11_log.txt`
4. 再对照 `Player.log`

原则非常简单：

- 先把实验性链路当作高风险层，而不是默认把锅丢给游戏本体。
- 先关闭这层复测一次，确认有没有直接关联。

### 4.5 防护、渠道、反作弊兼容相关问题

优先顺序：

1. 防护状态与错误提示
2. 应用运行日志
3. 游戏日志窗口
4. 再看游戏本体日志

因为这类问题很多是启动前就被阻断，游戏本体根本没有真正跑起来。

## 5. 怎样判断 warning 是噪音还是根因

不是所有 warning 都等价。

### 5.1 典型高噪音项

当前实现里，下面这些日志已经明确按“非致命噪音”处理：

- 可选 GStreamer 插件扫描失败
- `Couldn't get key from code: AudioVolumeUp`
- `Couldn't get key from code: AudioVolumeDown`

这类信息可以记下来，但不要默认把它们当成“掉帧、无响应、传送卡顿”的根因。

### 5.2 可能是症状，不一定是根因

这些内容要结合上下文判断：

- 单次出现的 `warning`
- SDK 网络探测失败
- WebView 超时
- 启动器层的驱动版本伪装信息
- 没有匹配到 ProtonFixes 全局规则

它们有时能解释“体验不完整”，但不一定解释“主线程长卡”。

### 5.3 值得优先追的信号

这些更值得优先追：

- 和时间点能精确对齐的重复异常
- 每次复现都稳定出现的相同调用栈
- 明确的 `error`、`fatal`、`panic`
- Unity 的长时间资源回收、对象扫描、资源卸载
- 防护检查失败、路径无效、主程序未找到、Prefix/运行器缺失
- 启用实验性功能后立即出现且可复现的问题

## 6. 一些高价值模式识别

### 6.1 `Player.log` 里出现长时间 `Unloading unused Assets`

这通常意味着：

- 场景切换或传送时发生了大规模资源回收；
- 真实卡顿点往往在 `MarkObjects` 这类对象扫描阶段；
- 这比单纯的“显卡没初始化”更像主线程长时间阻塞。

遇到这种情况，不要只盯着显卡驱动 warning，要先承认游戏本体自己就可能在做非常重的回收工作。

### 6.2 `PdhAddCounter`、`GetGpuMemory`、`OnlineProfilerDataManager`

这类模式常见于 Wine 下读取 Windows GPU 性能计数器失败。

它说明：

- 游戏正在尝试读取 GPU 显存或在线分析信息；
- Wine 兼容层可能拿不到对应计数器；
- 这往往是兼容性噪音或次级恶化因素，不一定是唯一根因；
- 但如果它反复出现并伴随异常上报，就值得记入问题描述。

### 6.3 `bridge-output.log` 有内容，但游戏日志窗口没有明显桥接结果

要考虑这些情况：

- 你没有在这一轮测试前打开“游戏日志窗口”；
- 游戏子进程还没退出，Bridge 回读尚未发生；
- 你需要直接打开 `bridge-output.log` 看原始内容。

### 6.4 没有 `d3d11_log.txt`

不要立刻下结论说“3DMigoto 没工作”。

先确认：

- 这轮有没有启用 3DMigoto；
- 当前 importer 和目标游戏是否匹配；
- 配置里是否开启了 3DMigoto 日志；
- 是否只是把日志功能关了。

## 7. 标准排查流程

下面这套顺序适合绝大多数复杂问题。

1. 先回到最小基线：关闭 3DMigoto、Bridge、Mods、ShaderFixes、自定义 DLL、自定义环境变量。
2. 保留当前 Proton、Prefix、DXVK、VKD3D，不要一次换太多层。
3. 打开“游戏日志窗口”，记录时间点并复现。
4. 先看启动链是否通。
5. 如果本体已经起来，再看 `Player.log` 是否存在明显主线程卡点。
6. 如果问题只在实验性功能开启后出现，再回到 Bridge / 3DMigoto 层。
7. 每次只改一项，再做下一轮复测。

如果你在第 1 步就不做最小基线，后面的很多日志会混在一起，很难得出可信结论。

## 8. 提交问题时至少要带什么

高质量反馈至少应包含这些内容：

- 游戏名称
- 问题现象
- 问题发生时间点
- 复现步骤
- 当前使用的 Proton/Wine
- 当前 Prefix、DXVK、VKD3D 状态
- 是否启用了 3DMigoto、Bridge、Mods、ShaderFixes、Jadeite
- `Player.log`
- 游戏日志窗口内容
- 如果启用了 Bridge，则附上 `bridge-output.log` 与 `bridge-config.json`
- 如果问题涉及启动器或外层链路，再附 `launcher.log`、`games.log`

缺少这些材料时，很多问题只能停留在猜测层。

## 9. 不建议的做法

- 没保留日志就开始清缓存、删 Prefix、重装运行环境。
- 还没确认根因，就同时切换多个 Proton 版本和多个图形层版本。
- 只看一条 warning，就宣称已经找到根因。
- 在启用 3DMigoto、Bridge、Mods 的同时，再去做运行环境大改。
- 用截图代替日志。

## 10. 推荐的长期工作流

如果你经常需要排查问题，建议形成固定习惯：

1. 先用稳定基线启动一次。
2. 需要实验性功能时，按 3DMigoto、Bridge、Mods 逐层加，不要一口气全开。
3. 每次改动前记下版本、运行器、Prefix 和关键开关。
4. 遇到异常先保留现场，再决定要不要做恢复操作。
5. 把 [已知问题与不足](06-已知问题与不足.md) 和各专题文档配合着看，不要只看这一篇。
