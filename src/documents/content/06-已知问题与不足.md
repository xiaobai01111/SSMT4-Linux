# 已知问题与不足

最后更新：2026-03-08

本文不是“简单缺陷列表”，而是帮助你理解这个项目当前真实边界的一份工程文档。它的目标不是制造悲观预期，而是让你知道哪些问题更像是：

- 使用方式不当；
- 上游环境不稳定；
- 预设或元数据缺失；
- 当前项目尚未覆盖；
- 或者本来就不应该由这个项目单独解决。

## 1. 先说结论

- 本项目不是“万能启动器”，而是一个围绕预设、运行环境、图形层、下载链和实验性扩展构建的管理工具。
- 某些能力已经比较完整，某些能力仍然是受支持范围、上游环境和元数据质量限制的工程方案。
- 用户在 UI 中看到的选项，不一定等于运行时最终生效的真实状态，因为启动器内部存在回退、修正和覆盖逻辑。
- 如果你把项目当作“无条件自动修复一切”的黑盒，几乎一定会在复杂问题上失望。

## 2. 支持范围不是无限的，而是预设驱动的

当前很多关键能力都不是“对所有游戏自动适配”，而是由游戏预设驱动。

这意味着：

- 自动下载不是所有游戏都有；
- 自动防护不是所有游戏都有；
- 渠道模式不是所有游戏都有；
- Jadeite 不是所有游戏都有；
- 主程序自动探测、启动链修正、默认运行环境、下载服务器、语音包等也都依赖预设。

所以“某个游戏没有某项功能”很多时候不是前端少了一个按钮，而是预设本身就没有定义这条链路。

## 3. 下载与修复能力当前存在明确边界

下载链已经比较完整，但“修复能力”并不统一。

当前已知边界包括：

- Snowbreak 暂不支持按异常列表做单文件修复，只能走完整下载或更新。
- HoYoverse 当前也不支持按异常列表单文件修复，异常文件通常需要重新下载。
- HoYoverse 下载严格依赖 `biz_prefix`，如果预设服务器配置缺少它，下载、校验和修复都无法正常继续。
- 并不是每个游戏都支持“官方启动器安装器模式”或“完整资源模式”两种形态。

因此：

- “支持下载”不等于“支持细粒度修复”；
- “支持校验”也不等于“校验失败后就一定能自动修复到位”。

## 4. 运行环境页的选择不等于最终启动命令

这是用户最容易误解的点之一。

当前项目里，运行环境页的设置更接近“偏好和约束”，不是对最终命令的绝对承诺。

已知行为包括：

- 开启 `umu-run` 但系统里没有 `umu-run` 时，会自动回退到 Proton 启动链。
- 开启 Pressure Vessel 但找不到 Steam Linux Runtime 时，会回退到直连 Proton。
- 某些预设会强制禁用 Pressure Vessel，覆盖你在 UI 里的选择。
- 启用 `umu-run` 时，额外的 `bwrap` 沙盒会被跳过，以避免容器嵌套冲突。

所以：

- UI 中勾选的 runner 不是“最终事实”；
- 真正的事实是启动器在那一轮解析之后实际用了什么。

## 5. 启动链比表面看起来复杂得多

项目当前支持的启动链并不只有“直接运行游戏 exe”这一种。

实际可能涉及：

- 启动器包装器
- 正式游戏主程序
- Bridge
- 3DMigoto
- Jadeite
- Proton / Wine runner
- umu-run / Pressure Vessel

这带来的已知问题是：

- 某一层配置错误，就可能让用户误以为“整个项目坏了”；
- 某些游戏要从 launcher exe 注入，再把真正的游戏 exe 作为目标进程；
- 某些游戏即使配置了包装器路径，启动时也会被纠偏到真正的主程序；
- 主程序路径检查有些只是提醒，不一定直接阻止启动，因此错误路径仍可能继续带来后续异常。

所以复杂问题不能只看“我点了开始游戏”这一层表象。

## 6. 3DMigoto、Bridge、Mods 仍然是实验性能力

这一点不只是法律风险声明，也是一条工程现实。

当前已知边界包括：

- 3DMigoto 依赖 Bridge 或相关桥接链，桥接程序不存在时会直接失败。
- 3DMigoto 受到全局开关和游戏级开关双重约束，不是只开一个地方就算启用。
- 某些 importer 会被预设自动修正，所以“你配置的 importer”不一定是最终使用的 importer。
- 这套链路会主动改变原始启动目标和注入逻辑，因此故障面天然大于纯启动器模式。

因此：

- 实验性功能更适合“验证、排查、定向使用”，不适合默认对所有游戏长期常开。

## 7. Jadeite 是特定游戏的补丁链，不是通用防护

Jadeite 当前属于特定游戏的反作弊兼容补丁，而不是通用能力。

已知边界包括：

- 它主要面向 HoYoverse 游戏，而不是任意预设。
- 如果本地没有 `jadeite.exe`，相关游戏可能根本无法正常启动。
- Jadeite 的下载和更新依赖远程 API，网络不可达、镜像异常或上游变更都会导致安装失败。
- Jadeite 与 3DMigoto 同时启用时，会进入更复杂的“Bridge + Jadeite + 注入”链路，排查难度更高。

所以：

- Jadeite 不是“性能选项”，也不是“普通工具包”，而是高风险兼容补丁。

## 8. 防护系统能做很多事，但也有明确边界

当前防护系统已经不只是“提示风险”，而是会真正改动环境：

- 改 `/etc/hosts`
- 删除预设定义的遥测 DLL
- 切换渠道配置字段
- 在启动前做强制或非强制检查

但它的已知边界同样很明确：

- 只有预设里定义了防护项的游戏，才有对应能力；
- “删除/禁用防护”不等于“完整恢复一切”，尤其渠道模式目前仍由单独入口控制；
- 已启用防护不等于实验性注入层也已安全；
- 防护系统不能替代账号安全判断，更不能承诺“不会封禁”。

所以不要把“防护已启用”误解成“可以随意叠加任何实验性功能”。

## 9. 图形与性能问题并不总能在 UI 层解释

图形和性能是当前最容易被误判的一类问题。

已知现实包括：

- 掉帧、卡顿、黑屏、无响应，很多时候来自 Wine/Proton、DXVK/VKD3D、驱动、Vulkan、GPU 选择、Unity 主线程阻塞或游戏自身资源管理。
- UI 里的 DXVK / VKD3D / Proton 开关可以缩小范围，但不能替代日志分析。
- 3DMigoto 启用后，会强烈改变图形链行为，问题责任层级也会随之变化。
- 某些启动期告警其实只是噪音，比如可选 GStreamer 插件缺失或输入映射告警，当前项目已把这类日志降级处理，不应默认当成根因。

所以“界面上看着没问题”不等于图形链一定正常。

## 10. 环境差异会带来高度不确定性

同一套配置在不同机器上可能有明显不同结果，这不是个别现象，而是兼容层项目的基本现实。

影响因素包括：

- 显卡品牌和驱动版本
- Vulkan 支持情况
- 桌面环境与显示协议
- 系统库版本
- Wine / Proton 版本
- Prefix 状态
- CPU / 存储性能
- 是否启用容器和沙盒

所以：

- 某个配置“在别人机器上能跑”不等于在你这里一定能跑；
- 某个版本“在你这里昨天能跑”也不保证今天换了运行环境还完全一样。

## 11. data-linux（旧名 Data-parameters）是关键依赖，不完整会影响很多功能

项目当前大量元数据依赖 `data-linux`（旧名 `Data-parameters`）。

已知边界包括：

- 仓库损坏或不完整时，项目会尝试 pull、修复工作区、重建或回退，但不保证每次都能无缝恢复。
- 缺少关键文件时，DXVK 变体、Proton catalog、部分预设和资源元数据会受到影响。
- 缺少对应 catalog 时，某些远程列表会回退到内置默认值或直接报错。
- 打包环境和开发环境的数据根路径不同，虽然有回退逻辑，但这本身就是一个潜在复杂度来源。

所以如果出现“版本列表异常”“预设丢失”“某些目录空了”，不要只盯着 UI，先确认 `data-linux` 状态。

## 12. 配置迁移仍然有降级路径，不是永远无痛

当前项目已经有 legacy 配置向 V2 配置的迁移逻辑，但并不意味着迁移总能完美成功。

已知边界包括：

- 迁移失败时，项目可能回退到 legacy 只读模式。
- 这时前端会显示只读告警，且保存接口会直接拒绝写入。
- 具体 warning code 例如 `MIGRATE_FAILED_READONLY`，它不是装饰信息，而是在告诉你“当前配置并不是正常可写状态”。

所以：

- 如果你发现某个配置“怎么改都不保存”，先不要急着怀疑 UI 卡了，先确认是不是掉进了只读降级路径。

## 13. 日志系统很重要，但也不是万能法官

日志是定位复杂问题的主要依据，但当前也有边界。

已知现实包括：

- 并不是所有外部组件都会输出足够详细的错误。
- 某些失败发生在上游运行时、被包装程序或外部依赖中，SSMT4 只能拿到“外层失败现象”。
- 部分问题需要把多个日志拼在一起看，单看一份日志容易得出错误结论。
- 有些日志是“高噪音低价值”，有些才是决定性的。

通常至少要结合这些材料：

- `Player.log`
- 启动日志
- 下载/防护日志
- Bridge / 3DMigoto 日志
- 当前运行环境和配置状态

## 14. 文档系统和本地文档也有边界

当前文档页优先使用随程序打包的本地 Markdown。

这带来两个现实：

- 离线可读性更好，打包后也能直接访问；
- 但本地文档的内容更新频率取决于当前程序版本，可能滞后于后续代码变更。

因此：

- 本地文档更适合作为“当前版本行为说明”；
- 在线 Wiki 更适合作为后续补充和增量说明。

## 15. 常见误区

- 误区一：支持下载就等于支持完整修复。不是，某些厂商链路目前仍只支持整包重下。
- 误区二：运行环境页选了某个 runner，就一定按那个 runner 启动。不是，内部存在多层回退和覆盖逻辑。
- 误区三：启用防护后，实验性注入也会一起变安全。不是，这是两套不同层级的系统。
- 误区四：有错误提示就一定是项目自身 bug。不是，很多错误是预设缺失、上游依赖缺失或环境不满足。
- 误区五：日志里出现的每个 warning 都值得优先处理。不是，一些高频启动噪音目前已经被明确降级。

## 16. 反馈问题时最常缺的关键信息

- 没有主程序绝对路径
- 没有游戏根目录和 Prefix 路径
- 没有 Wine / Proton / DXVK / VKD3D 版本
- 没有说明是否启用 3DMigoto、Bridge、Mods、ShaderFixes、Jadeite
- 没有说明是否应用了防护以及当前渠道模式
- 没有 `Player.log`、启动日志、Bridge 日志
- 没有复现步骤和问题发生时间点

如果这些信息缺失，很多问题根本无法判断责任层级。

## 17. 当前最推荐的使用策略

1. 先让游戏在“无实验性功能”的基础链路下能启动。
2. 再确认下载、防护、主程序路径和运行环境都正确。
3. 再确认 DXVK / VKD3D / Proton 组合稳定。
4. 最后才逐项开启 3DMigoto、Bridge、Mods、ShaderFixes、Jadeite 或自定义环境变量。
5. 每次只改一个变量，并保留一个可工作的基线配置。

如果你跳过前 4 步直接叠实验性层，那么“已知问题”会被你的人为放大成“所有东西都像坏了”。

## 18. 文档结论

你可以把当前项目的“已知问题与不足”理解成五类边界：

1. 支持范围边界：不是所有游戏、所有厂商链路、所有修复方式都已完整覆盖。
2. 运行时边界：UI 配置不等于最终命令，内部存在回退、覆盖和纠偏。
3. 上游依赖边界：Wine/Proton、驱动、Vulkan、data-linux、远程 API 都会影响结果。
4. 实验性功能边界：3DMigoto、Bridge、Mods、Jadeite 等能力天然更不稳定。
5. 诊断边界：很多复杂问题必须依靠多份日志和分层分析，不能只靠截图或单一现象判断。

所以这篇文档真正要提醒你的不是“问题很多”，而是：

- 不要把项目当成黑盒自动修复器；
- 不要把实验性功能当默认能力；
- 不要在缺少日志和基线的前提下同时改很多设置；
- 不要把所有异常都归咎于同一个开关。

如果你还没有读过 [项目风险与要求](01-项目风险与要求.md)、[游戏下载与主程序配置](02-游戏下载与主程序配置.md)、[防护与防封禁管理](05-防护与防封禁管理.md)，建议继续阅读。这三篇分别解释风险边界、启动链边界和防护边界。
